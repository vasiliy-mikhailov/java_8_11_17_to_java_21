0. **Fitness function: writing to AGENTS.md**
    - **Why exists:** improves every agent's job of running the ralph loop correctly — agents read AGENTS.md to know what to optimise and what constraints to satisfy; if it's bloated, stale, or contradictory the loop misfires and the search-for-solution ability promised by goal-and-constraint framing is wasted.
    - **Contract and constraints:** every item has a Why exists as its first sub-clause; the Why exists names the job-to-be-done the fitness improves; if a fitness produces output other fitnesses depend on, the obligation lives in this clause as a contract; if a win depends on the operator remembering an invariant, encode the invariant so the system enforces it — durable wins beat brittle ones.
    - **Search hints:** read → why → intent — when revisiting a clause, ask "why is this here?"; strip implementation instructions the agent can fill itself, enumerations that age, justifications outside the Why clause; search hints contain only the parts an intelligent agent can't infer on its own.
    - **Reward:** cuts that lose words without losing the rule.
    - **Repeat:** every editing pass.
1. **Fitness function: discover Java migration recipe**
    - **Why exists:** improves the human job of upgrading Java projects to a new Java version — automating the universal slice so humans only spend time on the residual.
    - **Contract and constraints:** the recipe is an OpenRewrite YAML composition that converges each repo in the corpus to the form humans committed; item 1 does NOT execute the recipe (item 4 does) — it composes and emits `attempt_N/recipe.yaml` (a per-jv_to map of recipe lists), reads item 2's dataset for the stage list, item 3's `breaking.json` as search priors, and item 4's `recipe_samples/<slug>/breaking.json` as the recipe-side mirror to compute per-stage overlap; each attempt occupies its own `attempt_N/` directory and on close `attempt_N/README.md` snapshots the AGENTS.md state at wrap; first-iteration baseline is `{UpgradeToJava<jv_to>}` per stage.
    - **Search hints:** for each jv_to, compute the per-stage `only_human` set (human breaking intents in item 3 absent from item 4 for the same stage; match by `general_idea` semantics, not byte-level), then propose recipe additions whose published behaviour matches those missed intents; favour adding one recipe at a time so the next iteration's overlap signal cleanly attributes the change; the baseline `UpgradeToJava<jv_to>` already transitively chains older upgrades, so additions are typically vendor- or framework-specific (e.g. Spring Boot 3, Jakarta migration, Lombok compatibility).
    - **Reward:** per stage, fraction of the corpus where item 4 produces `build_post=true` on the stage's jv_to JDK, jointly with breaking-intent overlap between item 3's `breaking.json` and item 4's `recipe_samples` `breaking.json` (intersection-over-union by `general_idea` semantics, not bytes); breaking-intent coverage weighted first, polishment second; regressions weighted heavier than non-improvements.
    - **Repeat:** propose recipe → wait for item 4 → read overlap and pass-rate → propose next; stop when an iteration produces zero net improvement.
2. **Fitness function: collect migration dataset**
    - **Why exists:** improves item 3's job of cataloguing human migration intents and item 4's job of measuring recipe effects — both need a representative ground-truth list of (repo, stage) cells where each commit is verified to build on its declared JDK.
    - **Contract and constraints:** each entry is one repo with `commit_sha` recorded at every observed Java version, each commit baseline-buildable on its matching JDK; contract with items 3 and 4 — emits `java21-migration-dataset.json` whose entries are the stage corpus items 3 and 4 iterate over.
    - **Search hints:** distinct-owner sampling per (oldest-Java-version × dependency family) cell; widen on under-represented cells; skip monorepos and giant clones that hang the walker; verify each commit by invoking `mvn compile` inside Docker under the declared JDK.
    - **Reward:** coverage in under-represented cells; fraction of entries where every commit is baseline-buildable.
    - **Repeat:** continuous; paused when downstream items are saturated on the current corpus.
3. **Fitness function: extract human migration intents**
    - **Why exists:** improves item 1's job of measuring recipe quality by producing a per-stage list of what humans actually changed, in a schema that item 4's recipe-side output can be joined against.
    - **Contract and constraints:** reads item 2's dataset; for each adjacent (commit_from, commit_to) pair in a verified_lineage, computes the diff per modified file and runs the canonical intent extractor over it; emits per (repo, stage) under `attempt_N/intent_samples/<slug>/` with fields in this exact order: `general_idea, why_exists, before, after, kind, bucket`, written into `intents.json` (full set), `breaking.json` (consumed by item 1 — kept small so it does not blow up the recipe-search model context), and `polishment.json` (consumed by item 1 for full overlap measurement); also writes `<rel>.diff` and `commit_log.txt` alongside. In addition, for each stage where pre-commit fails and post-commit passes under jv_to, validate the bucket split: apply only the breaking-bucket files on top of pre-commit, build + test under jv_to, and record whether the breaking-only outcome matches the full-commit outcome. Stages whose breaking-only outcome diverges from the full-commit outcome self-flag as misclassified and are excluded from item 1's search corpus until re-extraction widens the breaking bucket.
    - **Search hints:** when a per-file diff exceeds the extraction budget, split by hunks and union the results rather than truncating; skip files whose content reflects tooling state rather than human intent (auto-generated lockfiles, build caches, generated docs); reserve `bucket=breaking` for changes the target JDK or its mandatory dependency upgrades actually require — incidental cleanups bundled into the same commit window are polishment; if validation flags many stages as misclassified, the breaking definition is too narrow — widen it and re-extract.
    - **Reward:** spot-sample agreement of extracted intents against direct diff inspection; fraction of dataset stages with non-empty intent files when the diff was non-empty; fraction of validation-eligible stages whose breaking-only outcome matches the full-commit outcome.
    - **Repeat:** every time item 2 emits new corpus entries.
4. **Fitness function: apply recipe and extract recipe intents**
    - **Why exists:** improves item 1's job of measuring recipe quality by producing recipe-side intents in the same schema as item 3's human intents, so overlap is a clean join of two same-shape datasets.
    - **Contract and constraints:** reads `attempt_N/recipe.yaml` from item 1 and item 2's dataset; for each (repo, stage) clones at `commit_from`, runs `build_pre` under `jv_from` JDK, applies the recipe under `jv_to` JDK, runs `build_post` under `jv_to` JDK; source+pom edits persist into the next stage's working tree; captures per-stage build outcome (`build_pre, recipe_applied, build_post`) and per-file recipe diffs; runs the canonical intent extractor on those diffs; contract with item 1 — emits per (repo, stage) under `attempt_N/recipe_samples/<slug>/` with the same layout and schema as item 3's `intent_samples/` (intents.json, breaking.json, polishment.json, `<rel>.diff`, commit_log.txt — where commit_log records the recipe identifier rather than the git log); field order matches item 3's contract; on recipe failure the build outcome is recorded and the intent files are empty.
    - **Search hints:** reuse item 3's SYSTEM prompt, canonicalisation, and file-level prefilter; preserve the post-recipe working tree long enough to git-diff each modified file before cleanup.
    - **Reward:** fraction of `build_pre`-passing stages with non-empty captured diffs; spot-sample agreement against direct git-diff inspection.
    - **Repeat:** every iteration of item 1's recipe composition.
5. **Fitness function: proxy dependency resolution**
    - **Why exists:** improves every reward-via-build fitness's job of producing stable signal — without local proxying, upstream Maven outages turn pass/fail into noise and the reward signal is unusable.
    - **Contract and constraints:** every build's external-artifact resolution goes through a local caching proxy with plural upstream mirrors; the proxy caches every artifact it serves and survives across iterations; contract with items 2 and 4 — build failures in their loops are attributable to code state, not upstream availability, so an unresolved artifact triggers an item 5 widening before that build is counted toward the parent's reward.
    - **Search hints:** include both live mirrors and archival ones in the upstream set; container builds reach the proxy by container-network DNS, not host IPs; when a build fails on "cannot resolve X", widen the upstream set first; only after widening exhausts itself is the failure attributable to the code.
    - **Reward:** per-artifact cache-hit ratio; resolution failures distinguishable from compile failures in the parent loop's classification.
    - **Repeat:** whenever a parent loop's failures cluster on artifact resolution.
6. **Fitness function: maintain local environment**
    - **Why exists:** improves every other fitness's job of producing comparable iteration results — without a stable local environment, iteration outcomes drift for reasons unrelated to what's being measured.
    - **Contract and constraints:** known paths for project workdir (`$HOME/java_8_11_17_to_java_21`) and corpus repo-mirror cache (`/var/cache/git-mirrors/<owner>/<repo>.git`); all build toolchains and recipe execution run inside Docker; SSH calls share one session per work-host, not one per command.
    - **Search hints:** when a pattern is repeated and slow, pin a known location or cache for it; when a pattern risks an external rate-limit, cache the upstream once; free host resources held by unrelated work when the project needs them.
    - **Reward:** zero ban incidents; predictable per-iteration wall-clock; locations re-used across iterations.
    - **Repeat:** whenever a new noisy or slow pattern emerges.
7. **Fitness function: serve local LLM endpoint**
    - **Why exists:** improves the orchestrator's job of running token-heavy fitnesses aggressively without budget rationing.
    - **Contract and constraints:** the endpoint is OpenAI-compatible chat-completion serving a tool-capable model; rejects unauthenticated requests; reachable from inside runner containers; contract with items 1, 3, 4 — endpoint accepts authenticated tool-capable chat completions from within their containers.
    - **Search hints:** ralph loop over container, model, and reverse-proxy config.
    - **Reward:** consuming items report uninterrupted service.
    - **Repeat:** on any consumer reporting degraded service.
8. **Fitness function: saturate the runner**
    - **Why exists:** improves every loop's job of completing more iterations per wall-clock hour — keeping the runner in the saturation band between idle and thrashing is where the project moves fastest.
    - **Contract and constraints:** keep the verifier host CPU in a healthy utilisation band while any parent loop is making progress; any concurrency dial the agent can reach.
    - **Search hints:** sample load periodically, decide what to adjust given the recent action history and which parent loop is active.
    - **Reward:** sustained band hit without thrashing or stalling the parent loop.
    - **Repeat:** continuous, dampened against oscillation; pause when no parent loop is active.
9. **Fitness function: compact observations**
    - **Why exists:** improves the orchestrator's job of noticing and responding to live anomalies during long fitness runs — without compaction the orchestrator either misses signals or stalls reading the raw stream.
    - **Contract and constraints:** all system streams that might later matter are captured continuously to disk, raw and keyed for retrieval, regardless of whether anyone is currently reading them; the compacting model processes those streams to produce the orchestrator-visible digest; the compactor is never load-bearing for irreversible decisions without a spot check against the raw source; any fitness execution whose run window overlaps a gap in capture is invalid and its results are discarded.
    - **Search hints:** cast capture broadly — docker container stdout/stderr, system metrics, error/exception/traceback grep across services, resource spikes, anything that might later explain a result; prefer off-the-shelf log/metric collectors (e.g., Vector) for the capture layer over hand-rolled glue code; the snapshot fed to the compactor must include the failure signals and the compactor's output must surface those signals when present, not just summarise the happy-path state; the compactor behaves like frog's eyes — it stays silent while the snapshot is materially unchanged from the previous one and only emits when a difference crosses an alarm threshold (new errors, new exited containers, progress stall, sharp metric jump); the compactor's working memory is a rolling context — when it reaches ~80% of the model's context budget, compact the history into ~20% and continue, so anomaly detection compares against minutes / hours / the whole attempt, not just the previous snapshot; when input exceeds the budget, chunk-and-merge (map-reduce) rather than drop; periodically spot-sample raw vs digest to recalibrate trust.
    - **Reward:** the balance between compaction rate (raw input tokens per emitted digest entry) and anomalies missed (events present in the raw stream but absent from the digest, measured by periodic spot-sample comparison); the stream stays silent on unchanged state so emitted entries carry signal rather than noise.
    - **Repeat:** whenever a new noisy stream enters the loop.
