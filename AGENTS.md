# AGENTS.md

**Per-attempt history:** each `attempt_N/README.md` is a historical snapshot of the AGENTS.md state under which that attempt ran. It is not read by the agent — it exists only for audit.

1. **Dominanta (writing this file):** keep AGENTS.md compact and outcome-named.
   - **Purpose:** compact outcome-named files force the agent to re-derive the *how* each iteration from its tools and the corpus; without this discipline the file rots as the domain shifts and the agent over-fits to one iteration's specifics.
   - **Contract and constraints:** A dominanta in this file is a self-amplifying attentional and motivational attractor — an autonomous concern with its own single extremum (named in Reward) and its own trigger (named in Repeat) that pulls the agent's attention and self-amplifies until satisfied. Each dominanta is a concern the operator has offloaded from their own attention to the agent: the agent does the watching, the triggering, and the driving so the operator does not have to track it personally — this is what makes the file a delegation protocol, not a checklist. Only one dominanta is in foreground at a time (Ukhtomsky's principle: an active dominanta suppresses competing activity); the agent checks before every action that exactly one dominanta is currently dominant — concurrent dominantas are a contradiction in terms and indicate confusion that must be resolved before continuing. Control passes from one dominanta to another when a stronger trigger fires or the current extremum is approached; a dominanta interrupted mid-loop by a stronger trigger is resumed (not restarted) when control returns to it, so each dominanta persists enough state across switches (trajectory files, addenda, in-progress search position) for resumption to be genuine. A dominanta is written for an intelligent agent that searches for an implementation in a fuzzy environment, using the available tooling, resources, and harness in Ralph loops — so no implementation details should be supplied to the agent once it has demonstrated the ability to fill them in on its own. Dominantas are read by humans many times, so they must be kept as short as possible. Every dominanta in this file follows the same five-section structure as this one — Purpose, Contract and constraints, Search hints, Reward, Repeat. The Contract and constraints section is operator-only: the agent does not edit it; it is the immovable boundary within which the agent's other sections optimize. Section roles: Purpose names why the dominanta exists (would the project notice if it were removed?); Contract and constraints names the operator-defined boundary; Search hints record what observed failures have taught the agent to try first; Reward names a count, fraction, or directly comparable artefact the operator can read off without judgement, with one extremum per dominanta; Repeat names the trigger that brings the agent back to this dominanta. The dominanta one-liner names what the dominanta produces, not how — mechanism belongs inside Contract and constraints. Backtick-pinned concretes (paths, command shapes, file names, concrete element names) are scaffolding the reader uses to recognize the situation the rule applies to and stay through trims; enumerations that age (lists of versions, models, repo names) get stripped. AGENTS.md is organized in clusters: meta (Dominanta 1), the recipe and its experimental setup (Dominanta 2–3), and the substrate that lets the recipe loop run (Dominanta 4–9). Within a cluster, dominantas are ordered by what serves what.
   - **Search hints:** read → why → intent — when revisiting a clause, ask "why is this here?"; strip back to the rule when the answer is mechanism the agent already fills itself, a bare plea for the rule, or an aged enumeration; keep when the answer scopes when the rule applies, names a concrete that lets the reader recognize the situation, or signals when the rule should be revisited. A topic that always fires inside another dominanta's loop is not its own dominanta — it is mechanism of that loop, and belongs inside that loop's Contract.
   - **Reward:** cuts that lose words without losing the rule or its scope; every dominanta's Reward, after the pass, names something the operator can read off without judgement.
   - **Repeat:** every editing pass; additionally, whenever a dominanta's Contract and constraints is touched by the operator, the next pass over that dominanta re-audits its agent-mutable sections.

## Recipe cluster

2. **Dominanta (one-step Java LTS bump prompt + custom recipes):** for every (repo, jv_from → next LTS) bump where next LTS ∈ {11, 17, 21}, the paste-into-any-coding-agent prompt executed by the configured agent runtime (Dominanta 7) against the repo at `sha_from` produces a migrated tree that builds clean under the next LTS and preserves every test that was passing under `jv_from`. Full migration to Java 21 from earlier versions is a chain of these one-step bumps (8→11, 11→17, 17→21), each measured independently.
   - **Purpose:** an empirical migration recipe that beats the one-shot baseline is the primary research goal; the per-stage iterative loop is how the search converges across heterogeneous repos in the corpus.
   - **Contract and constraints:** the deliverable artifact is the prompt plus one operator-tested bump script per LTS transition (`bump_8_to_11.sh`, `bump_11_to_17.sh`, `bump_17_to_21.sh`) plus any custom claude-recipes the scripts invoke; the bump script is the canonical happy-path execution unit (the agent calls one shell command for the migration, does NOT re-derive the OpenRewrite recipe sequence); the agent only invents recovery on actual `[ERROR]` output, and recovery is bounded by a small table of scripted pom edits — self-contained for a stranger's repo, no references to our per-stage trajectory layout, no assumption of a specific backend or agent runtime; the same Dominanta 1 discipline applies (outcome-named, no instructions the agent fills itself, search hints derived only from observed failure patterns). A stage PASSes iff (a) `mvn compile` succeeds under `jv_to` AND (b) the set of `(test class, test method)` tuples that passed `mvn test` under `jv_from` before the agent's run is a subset of those that pass `mvn test` under `jv_to` after the agent's run; test results are read from `target/surefire-reports/TEST-<classname>.xml` (one XML per test class; multi-module projects glob recursively under `**/target/surefire-reports/TEST-*.xml`); integration tests run via maven-failsafe-plugin write to `target/failsafe-reports/` in the same XML format, but D2 measures unit tests via `mvn test`, which is surefire's domain; Gradle projects would write to `build/test-results/test/TEST-*.xml` but our corpus is Maven-only. `target/surefire-reports/` is cleared between pre and post runs so the two phases do not share state. empty pre-run pass set falls back to compile-only PASS so corpus repos without tests are not penalised. The agent operates inside git as the checkpoint mechanism — one recipe per iteration, `git reset --hard` on regression, `git commit` on success — so every change is reversible and observable; its action vocabulary is the recipe catalog the prompt names plus a small table of scripted fixes whose triggers must match the immediately preceding `[ERROR]` verbatim. The agent does not author bespoke source or pom edits outside this vocabulary. Writes per-repo trajectories under `attempt_N/per_repo_iter/<slug>/trajectory.json` (agent runtime + backend model + prompt fingerprint, event-stream summary, verdict, failure observation, wall-clock, pre/post test counts, migrated-tree diff vs `sha_from`); reads Dominanta 3's dataset for the stage list; reads Dominanta 5's host paths; consumes Dominanta 4, 6, 7, 8, 9 transparently. Per-repo search is iterative — run the prompt under the configured agent runtime against the current working tree, on FAIL append the `[ERROR]` block + maven summary + the agent's own session summary into `observation_addendum.md` (per-stage operator-authored enrichment that rides on the in-harness invocation as the authoritative override for that stage, overriding generic search hints for the same failure cluster; for paste-into-agent use, the operator hand-edits the prompt instead), retry; a session whose action fingerprint matches one already tried bails. Agent runtime, backend model, and prompt version are pinned per attempt so PASS rate is comparable across passes. The corpus is processed as a multi-pass round-robin — each pass gives every still-FAILing repo K more attempts continuing its trajectory, then moves on; iterate passes until one yields zero new PASSes. Within a pass, up to N stages are dispatched concurrently to Dominanta 7's agent runtime, where N is the per-loop concurrency dial Dominanta 8 currently advises; each concurrent session is isolated (per-stage workspace, per-stage trajectory file).
   - **Search hints:** craft the prompt the way AGENTS.md is crafted (see Dominanta 1); never restate what the agent does well naturally; pair every hint with the observed failure mode that motivates it and prune the hint once that failure stops recurring; keep worked examples when trimming generic instruction — examples are scaffolding the agent imitates, and the abstract rule alone is weaker. When the agent applies the right step shape but consistently picks the wrong target, the gap is information-availability not hint quality — enrich `observation_addendum.md` with the specific pom element / classpath dump / source-grep result the agent needs, rather than enumerating possible targets in the prompt. Persist regressed-test list into the trajectory so the agent sees which specific tests broke and the operator can spot-check. Regime change: when the harness, recipe artifact set, or environment is altered in a way that materially changes what counts as a failure, INVALIDATE the trajectories of every stage whose history was conditioned on the old regime; cheapest invalidation is deleting the affected trajectory files so the next pass starts those stages from scratch. When the agent cannot reach a fix regardless of prompt or observation enrichment (requires a primitive that doesn't exist), ship that primitive deterministically and surface it in the prompt, not more hints. Per-step log dirs prevent successive same-named phases from overwriting each other; default K=1 — pass N grants budget=N attempts per still-FAILing stage; trajectory state persists between passes so each new attempt picks up prior session summary. Watch action count per step on smoke runs: if any single discrete step (baseline capture, recipe apply, scripted fix application) takes more than 5 agent actions, the prompt is missing a specific hint for that step — patch the prompt with the canonical mechanism before scaling.
   - **Reward:** corpus PASS rate (as defined in Contract: compile + test-conservation) vs the one-shot `UpgradeToJava<jv_to>` baseline.
   - **Repeat:** continuous round-robin until convergence; restart from cached trajectories whenever the prompt changes materially.
3. **Dominanta (dataset rediscovery):** curate `java21-migration-dataset.json` as distinct-owner lineage samples balanced per (oldest-Java-version × dependency family) cell.
   - **Purpose:** a corpus biased toward already-prep'd states would let Dominanta 2 hit high PASS rates without doing real migration work; the dataset must sample real migration windows so the recipe's value is measurable.
   - **Contract and constraints:** emits `java21-migration-dataset.json`; Dominanta 2 reads it for the stage list and excludes any stage whose `sha_from` either fails `mvn compile` under jv_from or has zero passing tests (which would make Dominanta 2's test-conservation criterion degenerate). Each entry is one (repo, jv_from→jv_to) stage; `sha_from` is picked by starting from `next_version_commit` (the first commit at the next Java version in the repo's lineage), walking back N commits along `--first-parent`, and accepting a commit that compiles AND whose unit-test suite passes a meaningful majority of tests under jv_from. `sha_to` is the version-bump commit at the target Java version. A full-lineage J8→J11→J17→J21 repo yields THREE stages (J8→J21, J11→J21, J17→J21), each using the same `sha_to` (the J21-bump) and a different commit-distance-back `sha_from` from the relevant intermediate-bump. Each entry records `pre_pass_count` (passing-test cardinality at `sha_from`) and `commits_back` (distance from `next_version_commit`) so Dominanta 2 can weight stages.
   - **Search hints:** distinct-owner sampling; widen under-represented cells; prefer full-lineage repos over partial-lineage when both fit a cell; skip monorepos and giant clones that hang the walker; accept a candidate `sha_from` iff it compiles under jv_from AND passes a soft majority (default ≥0.7) of its non-skipped tests; probe at increasing commit-distances back (start 20, then 50, 100, 200) and accept the smallest distance meeting the bar; flag any stage whose `sha_from`→`sha_to` diff touches only the Java-version property or compiler-plugin source/target/release knob — the step-back wasn't deep enough and the stage is measuring property mutation, not recipe quality. When full-lineage repos are scarce, fall back to single-step (jv_from → jv_from+1) stages that nevertheless exercise breaking changes (Spring Boot 2.x→3.x, javax→jakarta, Hibernate 5→6, JUnit 4→5) — a small corpus rich in framework-migration diversity is more valuable than a large corpus of trivial property-flip migrations.
   - **Reward:** fraction of (oldest-Java-version × dependency-family) cells with ≥1 stage where `sha_from` carries ≥1 passing test (so Dominanta 2's criterion is enforceable, not skipped via empty-pre fallback) and the `sha_from`→`sha_to` diff exercises a real breaking change.
   - **Repeat:** continuous; paused when Dominanta 2 is saturated on the current corpus.

## Substrate cluster

4. **Dominanta (proxy dependency resolution):** every build's external-artifact resolution goes through a local caching proxy with plural upstream mirrors.
   - **Purpose:** without a caching proxy, build failures get attributed to upstream availability blips rather than code state, polluting the parent loop's reward signal.
   - **Contract and constraints:** the proxy caches every artifact it serves and survives across iterations. Consumers (Dominanta 2, Dominanta 3) attribute build failures to code state rather than upstream availability, so an unresolved artifact triggers a proxy widening before that build is counted toward the parent's reward.
   - **Search hints:** include both live mirrors and archival ones in the upstream set; container builds reach the proxy by container-network DNS, not host IPs; when a build fails on "cannot resolve X", widen the upstream set before treating the failure as code-attributable.
   - **Reward:** per-artifact cache-hit ratio (higher is better; resolution failures distinguishable from compile failures in the parent loop's classification follows from this).
   - **Repeat:** whenever a parent loop's failures cluster on artifact resolution.
5. **Dominanta (maintain local environment):** known stable paths, scratch-space discipline, and disk/cache stewardship for every other dominanta.
   - **Purpose:** iteration outcomes must vary only with what's being measured; unstable paths, shared scratch space, and resource contention drift the measurement noise floor up and obscure the signal; one dominanta exhausting a shared resource crashes another and corrupts the parent loop's results.
   - **Contract and constraints:** project workdir at `$HOME/java_8_11_17_to_java_21` (the one location pinned in this dominanta — every other path is an env var resolved from `.env` so the value lives next to credentials). Env vars consumers read: `MIRROR_CACHE_DIR` (per-repo git mirrors live as `<owner>/<repo>.git` under it); `SCRATCH_ROOTS` (colon-separated writable mounts ordered by preference; agent picks the first with sufficient headroom); `SCRATCH_SUBDIR_TEMPLATE` (relative template expanded per scratch dir, e.g. `tmp/<repo_name>/<ff_specific_subdir>/`). All build toolchains and recipe execution run inside Docker; SSH calls to the work host share one session, not one per command; free host resources held by unrelated work when this project needs them. Individual dominantas declare a scratch-dir name; this dominanta resolves it against `SCRATCH_ROOTS` + `SCRATCH_SUBDIR_TEMPLATE`. Stale per-iteration dirs older than N hours and orphaned (no live process holding them) are reaped continuously, not on demand.
   - **Search hints:** when a pattern repeats and is slow, pin a known location or cache for it; when a pattern risks an external rate-limit, cache the upstream once; disk pressure (any mount over a threshold) triggers immediate janitor pass over the project's scratch root; never let docker-owned root-rw files leak — clean them through a busybox container, not host `rm`.
   - **Reward:** iteration outcome variance from non-measured factors approaches zero — observable as zero ban incidents, predictable per-iteration wall-clock, locations re-used across iterations, zero disk-full crashes during any parent loop, bounded scratch-dir count per repo (a leak indicator triggers investigation, not just cleanup).
   - **Repeat:** whenever a new noisy or slow pattern emerges, or whenever the previous run leaked scratch dirs beyond the bounded budget.
6. **Dominanta (vLLM spin-up):** stand up tool-capable OpenAI-compatible chat-completion endpoints callable from inside Docker, with authentication enforced.
   - **Purpose:** Dominanta 2's agent backend and Dominanta 9's compactor cannot run without a tool-capable model endpoint reachable from inside docker containers; this dominanta owns that contract surface.
   - **Contract and constraints:** all endpoints in active use serve models from the same generation family — never mix generations, since prompts calibrated against one generation drift on another. When two consumers have meaningfully different throughput / depth tradeoffs they get separate endpoints on separate accelerators so they don't contend for slots. Per-consumer model identity and call-time options (thinking mode, etc.) live in the consumer's own Contract.
   - **Search hints:** arrive at a working endpoint set through a ralph loop over container choice, model choice, and reverse-proxy config; converge by what the consumers can actually call, not what the endpoint exposes in isolation.
   - **Reward:** consumers report uninterrupted service.
   - **Repeat:** on any consumer reporting degraded service.
7. **Dominanta (agent runtime spin-up):** stand up the configured agent runtime as a credentialed service reachable from the work host, with conversation lifecycle, event-stream emission, and context-management primitive configured.
   - **Purpose:** Dominanta 2's recipe search cannot run without an agent runtime that accepts a prompt, executes against a working tree, emits structured events the harness can consume, and survives the deepest stages without losing trajectories to context overflow; this dominanta owns that contract surface.
   - **Contract and constraints:** the runtime exposes a callable interface (in-process SDK, HTTP API, or MCP socket) reachable from the harness; supports tool invocation, conversation state across iterations, and event streaming. The runtime is configured with at least one context-management primitive — an automatic conversation condenser that folds older events into a summary at a configured threshold, or the ability to spawn delegated subagents with their own context windows — so the agent's single-call context budget does not scale with repo depth and overflow is structural, not catastrophic. Every exposed surface (server endpoint, conversation API, MCP socket, container exec interface) shares the credential discipline of Dominanta 6's vLLM endpoints — no unauthenticated bind, no credentials in trajectory state, no credentials in scratch dirs reaped by Dominanta 5. Sub-context activity is captured in the event stream (Dominanta 9) so the operator can audit when and how context was folded. The runtime has network reach to Dominanta 4's caching proxy (same docker network or equivalent route) so every Maven invocation the agent makes resolves artifacts through the cache, not direct upstream — without this guarantee, parent-loop wall-clock balloons and upstream hosts ban us for download volume. Runtime identity is pinned per attempt so Dominanta 2's PASS rates are comparable across passes. The runtime accepts N concurrent isolated sessions (each with its own conversation id, workspace path, and event-stream slice) where N is dictated by Dominanta 8; sessions share the LLM endpoint and the Nexus proxy but never share a workspace.
   - **Search hints:** prefer maintained frameworks over reinventing tool-call iteration, context pruning, or conversation state; choose a runtime by what Dominanta 2 can actually drive, not by what the runtime advertises. When the runtime needs network reach (server mode, MCP), it goes through the same reverse-proxy + auth surface as the vLLM endpoints; in-process SDK use is permitted but does not exempt the runtime from the credential discipline of any external endpoint it dials. If the agent stalls or repeats actions on the same failure across iterations, the missing input is usually structural source detail that needs sub-context — not a different prompt.
   - **Reward:** Dominanta 2 reports uninterrupted agent execution; fraction of stages where context overflow caused trajectory loss approaches zero; zero credentialless surfaces in agent-runtime configuration.
   - **Repeat:** on any Dominanta 2 stage reporting an agent runtime failure unrelated to the prompt or repo state; whenever context overflow drops a trajectory; whenever a stage cluster shows the agent stalling without obvious cause.
8. **Dominanta (runner saturation):** keep every host compute resource (CPU, memory, GPU compute, GPU memory, disk I/O, network) in a healthy utilisation band while any parent loop is making progress; the binding constraint shifts across the loop's phases and is re-detected, not assumed.
   - **Purpose:** an idle host wastes budget; any over-saturated resource drops the parent loop's wall-clock-per-PASS reward to noise. The binding constraint is rarely CPU alone — under N-way agent parallelism it commonly shifts from CPU at startup to GPU compute during agent rounds to disk/network during mvn cache warming to GPU KV-cache memory under sustained batching.
   - **Contract and constraints:** composes with the parent loop rather than standing alone — saturation decisions are made conditional on which loop is active, what its recent action history is, and which resource is currently nearest its band edge; any concurrency dial the agent can reach is in scope. The primary output is a per-loop concurrency recommendation N (read by Dominanta 2 to size its per-pass session fan-out into Dominanta 7's runtime); N may shift mid-pass as the binding constraint shifts.
   - **Search hints:** sample every relevant resource periodically (host CPU + load avg, host memory, per-GPU utilisation + memory + KV-cache usage from the vLLM logs, disk I/O on the work mount, network to Dominanta 4's proxy and Dominanta 6's LLM endpoint, docker daemon health); learn the band per resource; re-detect the binding constraint when the parent loop changes phase. GPU compute-percent snapshots are bursty under request batching — read average over a window, not instantaneous.
   - **Reward:** sustained healthy band across every relevant resource without thrashing or stalling the parent loop; the binding constraint correctly identified at any moment.
   - **Repeat:** continuous, dampened against oscillation; pause when no parent loop is active.
9. **Dominanta (compact observations):** capture system streams continuously to disk and run a compacting model over them to produce the orchestrator-visible digest.
   - **Purpose:** raw observation streams exceed the orchestrator's context budget; without a faithful digest, decisions are made from the wrong fraction of the data and the operator's trust in the digest erodes silently.
   - **Contract and constraints:** the compactor model is Qwen 3.6 27B AWQ (dense, served from Dominanta 6), called with `chat_template_kwargs.enable_thinking: true`; fall back to no-thinking only when thinking output is unparseable. Capture runs regardless of whether anyone is reading it (raw streams keyed for retrieval); the compactor is never load-bearing for irreversible decisions without a spot check against the raw source. The digest is multi-resolution — at least four logarithmically spaced windows (currently `10s / 60s / 10m / 60m`) are sampled side-by-side, so anomalies are spotted by window-to-window contrast rather than from a single cut; gauges report first/last/min/max per window and counters report deltas, so adjacent windows differ on slow-moving signals instead of looking frozen. The captured stream set includes the agent runtime event stream (Dominanta 7, summary-only so the compactor's context budget does not scale with investigator depth) alongside Vector's docker / app_logs / host_metrics. Any dominanta execution whose run window overlaps a gap in capture is invalid.
   - **Search hints:** Vector is the canonical capture layer; it already runs as the `obs-vector` container and sinks observed streams as JSONL to `/var/log/observe/` (`docker.jsonl`, `host_metrics.jsonl`, `app_logs.jsonl`); the compactor reads tails of those JSONL files rather than polling subprocesses; new observable streams enter the loop by being added to `/tmp/vector.toml`, not by extending the compactor. The snapshot fed to the compactor must surface failure signals when present, not just happy-path state; emit only when state changes materially since the previous snapshot, so a quiet stream stays quiet; on input exceeding the compactor's budget, chunk-and-merge rather than drop; periodically spot-sample raw vs digest to recalibrate trust.
   - **Reward:** balance between compaction rate and anomalies missed; quiet streams stay quiet so emitted entries carry signal.
   - **Repeat:** whenever a new noisy stream enters the loop.
