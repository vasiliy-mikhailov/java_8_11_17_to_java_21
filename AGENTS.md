# AGENTS.md

**Per-attempt history:** each `attempt_N/README.md` snapshots the AGENTS.md state that attempt ran under — audit only, not read by the agent.

1. **Problem (writing this file):** keep AGENTS.md compact and outcome-named.
   - **Value:** compact, outcome-named files force the agent to re-derive the *how* each iteration from its tools and the corpus; without this discipline the file rots as the domain shifts and the agent over-fits to one iteration's specifics.
   - **Contract and constraints:** A problem is a self-amplifying attractor — one autonomous concern with a single extremum (its Reward) and a single trigger (its Attention mechanism) that pulls the agent's attention and amplifies until satisfied. Each is a concern the operator has offloaded to the agent (the agent watches, triggers, and drives it), making this file a delegation protocol, not a checklist. Only one problem is in foreground at a time (Ukhtomsky); before every action the agent verifies exactly one is dominant — concurrent problems signal confusion to resolve first. Control passes when a stronger trigger fires or the current extremum is approached; an interrupted problem is resumed, not restarted, so each persists enough state (trajectory files, addenda, in-progress search position) for resumption to be genuine. Problems are written for an intelligent agent searching a fuzzy environment with the available tooling in Ralph loops: supply no implementation detail the agent can fill itself, and — since humans reread this file often — keep each as short as possible. Every problem has five sections: **Value** (why it exists — would the project notice its removal?); **Contract and constraints** (the operator-only boundary the other sections optimize within — the agent does not edit it); **Solution search approach and hints** (what observed failures taught the agent to try first, plus this agent's own loop or trigger for returning to the problem); **Reward** (one extremum the operator reads off without judgement — a count, fraction, or comparable artefact); **Attention mechanism** (how a change this problem makes or notices in the shared environment becomes salient to whichever problem must react — the channel it publishes a signal to or watches, e.g. a log line, a metric, a `feedback.jsonl` entry, or an observability stream — as distinct from the agent's own re-entry loop). The one-liner names what the problem produces, not how. Backtick-pinned concretes (paths, command shapes, file and element names) are recognition scaffolding and survive trims; aging enumerations (versions, models, repo names) get stripped. The file is clustered — meta (D1); recipe and setup (D2–D4); substrate (D5–D10) — and within a cluster ordered by what serves what.
   - **Solution search approach and hints:** read → why → intent — for each clause ask "why is this here?". Strip when the answer is mechanism the agent already fills, a bare plea for the rule, or an aged enumeration; keep when it scopes when the rule applies, names a recognition concrete, or signals when to revisit. A topic that always fires inside another problem's loop is mechanism of that loop — it belongs in that loop's Contract, not as its own problem. Re-run every editing pass; additionally, whenever the operator touches a problem's Contract and constraints, the next pass re-audits that problem's agent-mutable sections.
   - **Reward:** cuts that lose words without losing the rule or its scope; after the pass, every problem's Reward names something the operator can read off without judgement.
   - **Attention mechanism:** the file is the channel — an operator edit to a problem's Contract, or a problem's agent-mutable sections visibly bloating, is the signal that pulls this meta-problem to re-audit that problem.

## Recipe cluster

2. **Problem (Java LTS bump artifact for any coding agent):** ship one `(prompt.md, recipe catalog)` artifact — consumable by any tool-using coding agent (no assumption of our harness, scratch layout, or runtime) — that on each next-LTS bump (8→11, 11→17, 17→21) drives the highest corpus PASS rate.
   - **Value:** an empirical migration artifact that beats the one-shot OpenRewrite baseline is the primary research goal; portability across agents is the audience constraint, so the deliverable is self-contained and harness-agnostic.
   - **Contract and constraints:** the artifact is `(prompt.md, recipe catalog)` — prompt shape is the six-section form D3 pins, recipe identities are FQNs in a versioned catalog, both stage-agnostic (stage facts arrive via the consumer's harness as a prepended Stage header). Reads D4 for the stage list. Per-stage validation is delegated to D3, which returns `{terminal_rung, terminal_outcome, artifact_updates}`; fold `artifact_updates` into the corpus artifact iff that does not regress corpus reward, else leave it unchanged and record the stage unintegrated. Concurrency across stages from D9.
   - **Solution search approach and hints:** when a failure family's stages only pass at the strongest rung (Opus) and won't harden down to OH+Qwen, that family lacks a corpus-level remedy — commission a recipe via D3's search or add a failure-output row. An `artifact_updates` that passes its own stage but regresses others on rollout is over-fit — generalise its trigger or scope the fix narrower. Loop continuously round-robin until reward stops improving, restarting from cached trajectories whenever the corpus artifact's sha materially changes.
   - **Reward:** corpus PASS rate (fraction of stages D3 terminates at the OH+Qwen rung with `terminal_outcome=PASS`) vs the one-shot `UpgradeToJava<jv_to>` baseline.
   - **Attention mechanism:** `feedback.jsonl` is the channel — a failure family clustering there, or its stages only passing at the strong end of the rung ladder (stuck at Opus), signals that the family needs a corpus-level remedy.
3. **Problem (per-repo prompt + recipe escalation):** for each (repo, commit) stage D2 hands down, harden the current best (prompt, recipes) artifact until it drives the bump to PASS or labelled BAIL under OpenHands+Qwen — by making the stage pass down a ladder of progressively weaker executors, all driven by Claude: first Claude+Opus 4.8, then Claude+Qwen, then Claude managing OpenHands+Qwen. Claude patches the artifact at each rung so it survives the next, weaker one.
   - **Value:** D2 sees corpus PASS/FAIL rates but cannot distinguish "stage genuinely unfixable", "production LLM too weak", "production harness wrong", and "artifact has a gap". This problem separates those for one stage: the weakest rung at which the artifact still passes names the failure mode — fails even at Opus → genuinely unfixable or needs a new recipe; passes at Opus but not Qwen → unclear to a weak LLM; passes at Qwen but not OH+Qwen → incompatible with OH's tool-use conventions — so D2 acts on signal, not noise. It is also the path by which a single stage's prompt/recipe insight propagates into the corpus artifact.
   - **Contract and constraints:** The (prompt, recipes) artifact has a pinned shape so every rung reads the same structure and a failure maps to a known section. The prompt is one file whose six sections — Purpose, Contract and constraints, Basic flow, Failure outputs → remedies, Reward, When to bail — are pinned in this order (the live instance is `corpus_best/prompt.md`). Two pins a reader needs here: the Failure-outputs table's fix vocabulary is {pom edit, `mvn rewrite:run` by FQN, docker cleanup, labelled bail, ignore-with-justification}, and the Reward (PASS) is `mvn compile` succeeds under jv_to AND pre-pass test set ⊆ post-pass test set. Recipes are whatever the table names by FQN; they live in the cache pool the bump scripts pull from, with identities (not source) pinned per artifact revision. Stage facts (repo URL, sha, jv_from, jv_to, workdir) are injected by each rung's harness as a prepended Stage header, never in the prompt, so the artifact stays portable. The search is one schema: per stage, the **search results** — the artifact version run at each rung (`opus`, `qwen`, `oh_qwen`) with its outcome — and the one **best search result**, the hardened artifact that reaches the weakest rung (OH+Qwen). Claude is the sole searcher at every rung: it runs the current artifact under that rung's executor and, on failure, patches the artifact (rewrite any prompt section, add or reorder failure rows, add recipe FQNs publishing to the catalog, edit bump scripts); the executor (Opus 4.8, Qwen 27B FP8, or OpenHands+Qwen 27B FP8) only runs what Claude ships. The ladder goes strong → weak because each weaker rung exposes a gap the stronger one hid — Opus surfaces "genuinely unfixable / needs a new recipe", Qwen surfaces "unclear to a weak LLM", OH+Qwen (the production target) surfaces "incompatible with OH's tool-use conventions"; a fix for one rung is re-validated at the rung below, since it can regress a stronger one. A rung's FAIL is re-run-confirmed (flakes re-run at the same rung, not escalated); the stage terminates when the OH+Qwen rung reaches PASS or labelled BAIL. Per (stage × rung × validation-run) the harness appends one line to `attempt_N/feedback.jsonl` (`rung ∈ {opus, qwen, oh_qwen}`, `outcome ∈ {PASS, BAIL:<label>, FAIL}`) and, before invoking a rung, snapshots the exact prompt to `prompt_snapshots/<sha>.md` and catalog to `recipe_snapshots/<sha>.jar` (content-addressed, idempotent); the full per-stage record is `per_repo_iter/<slug>/trajectory.json` alongside `oh_dialogue.log` and `bump.diff`. Each rung's session is isolated (own workdir, conversation id, surefire path); concurrency bounded by D9. Output to D2: `{stage, terminal_rung, terminal_outcome, artifact_returned}`, where `terminal_rung` is the weakest rung reached at PASS or labelled BAIL (ideally `oh_qwen`) and `artifact_returned` is the hardened artifact; D2 decides folding.
   - **Solution search approach and hints:** fast-fail at a rung — if Claude+executor stalls (no progress for >N actions on the same step) treat it as FAIL at that rung, don't spin; never recurse on flakes (one-off network error, Docker glitch) — re-run the same rung. On a FAIL, Claude's first probe is to read the executor's actual trace (`oh_dialogue.log`) to find where its reasoning diverged from the prompt — typically a missed trigger paraphrase, a skipped step, or a tool-use convention the prompt didn't anticipate. Claude's patch is the smallest possible (a one-sentence tightening, a row reword, a step reorder), not a wholesale rewrite; add a new recipe only when no prose change could fix the executor's behaviour. After patching for one rung, re-validate at the rung below, since a fix can regress a stronger one. A stage that keeps failing at the OH+Qwen rung even after patching means the OH harness has a structural limitation the artifact form cannot route around — surface that to D8, not as a prompt rewrite. Re-invoke per stage each pass from D2, and again when the corpus artifact's sha changes and the stage previously did not reach PASS at the OH+Qwen rung.
   - **Reward:** fraction of input stages where the OH+Qwen rung reaches PASS or labelled BAIL — both initially and after each patch; the metric is "OH+Qwen-rung success rate", not "any-rung success rate". Secondary: average patches per stage to reach OH+Qwen success (lower is better).
   - **Attention mechanism:** `feedback.jsonl` is the channel — a re-run-confirmed FAIL trajectory written there is the signal that pulls Claude to patch the artifact at that rung; an OH+Qwen rung still failing after patches signals D8 (a structural harness limit), not another prompt rewrite.
4. **Problem (dataset rediscovery):** curate `java21-migration-dataset.json` as distinct-owner lineage samples balanced per (oldest-Java-version × dependency family) cell.
   - **Value:** a corpus biased toward already-prep'd states would let D2 hit high PASS rates without doing real migration work; the dataset must sample real migration windows so the recipe's value is measurable.
   - **Contract and constraints:** emits `java21-migration-dataset.json`; D2 reads it for the stage list and excludes any stage whose `sha_from` fails `mvn compile` under jv_from or has zero passing tests (which degenerates D2's test-conservation criterion). Each entry is one (repo, jv_from→jv_to) stage; `sha_from` starts from `next_version_commit` (first commit at the next Java version in the lineage), walks back N commits along `--first-parent`, and is accepted when it compiles AND its unit-test suite passes a meaningful majority under jv_from; `sha_to` is the version-bump commit at the target Java version. A full-lineage J8→J11→J17→J21 repo yields three stages sharing `sha_to` with different commit-distance-back `sha_from`s. Each entry records `pre_pass_count` and `commits_back` so D2 can weight stages.
   - **Solution search approach and hints:** distinct-owner sampling; widen under-represented cells; prefer full-lineage over partial-lineage when both fit a cell; skip monorepos and giant clones that hang the walker; accept a `sha_from` iff it compiles under jv_from AND passes a soft majority (default ≥0.7) of non-skipped tests; probe at increasing commit-distances (20, 50, 100, 200) and accept the smallest meeting the bar; flag any stage whose `sha_from`→`sha_to` diff touches only the Java-version property or compiler source/target/release knob — the step-back wasn't deep enough and the stage measures property mutation, not recipe quality. When full-lineage repos are scarce, fall back to single-step stages that still exercise breaking changes (Spring Boot 2.x→3.x, javax→jakarta, Hibernate 5→6, JUnit 4→5) — diversity beats a large corpus of trivial property-flips. Run continuously; pause when D2 is saturated on the current corpus.
   - **Reward:** fraction of (oldest-Java-version × dependency-family) cells with ≥1 stage where `sha_from` carries ≥1 passing test (so D2's criterion is enforceable) and the `sha_from`→`sha_to` diff exercises a real breaking change.
   - **Attention mechanism:** D2's exclusions and cell coverage are the channel — a (Java-version × dependency-family) cell with no enforceable stage, or D2 saturating the current corpus, signals D4 to widen sampling there.

## Substrate cluster

5. **Problem (proxy dependency resolution):** every build's external-artifact resolution goes through a local caching proxy with plural upstream mirrors.
   - **Value:** without a caching proxy, build failures get attributed to upstream availability blips rather than code state, polluting the parent loop's reward signal.
   - **Contract and constraints:** the proxy caches every artifact it serves and survives across iterations. Consumers (D2, D4) attribute build failures to code state, so an unresolved artifact triggers a proxy widening before that build counts toward reward.
   - **Solution search approach and hints:** include both live and archival mirrors; container builds reach the proxy by container-network DNS, not host IPs; when a build fails on "cannot resolve X", widen the upstream set before treating the failure as code-attributable. Re-enter whenever a parent loop's failures cluster on artifact resolution.
   - **Reward:** per-artifact cache-hit ratio (higher is better; resolution failures become distinguishable from compile failures in the parent loop's classification).
   - **Attention mechanism:** build logs are the channel — a `cannot resolve X` line is the signal that pulls D5 to widen the upstream set before that failure is counted as code-attributable.
6. **Problem (maintain local environment):** known stable paths, scratch-space discipline, and disk/cache stewardship for every other problem.
   - **Value:** iteration outcomes must vary only with what's being measured; unstable paths, shared scratch, and contention drift the noise floor up, and one problem exhausting a shared resource crashes another and corrupts the parent loop's results.
   - **Contract and constraints:** project workdir at `$HOME/java_8_11_17_to_java_21` (the one path pinned here — every other is an env var resolved from `.env` so the value lives next to credentials). Env vars consumers read: `MIRROR_CACHE_DIR` (per-repo git mirrors as `<owner>/<repo>.git`); `SCRATCH_ROOTS` (colon-separated writable mounts by preference; agent picks the first with headroom); `SCRATCH_SUBDIR_TEMPLATE` (relative template per scratch dir, e.g. `tmp/<repo_name>/<ff_specific_subdir>/`). All build toolchains and recipe execution run inside Docker; SSH calls to the work host share one session, not one per command; free host resources held by unrelated work when this project needs them. A problem declares a scratch-dir name; this problem resolves it against `SCRATCH_ROOTS` + `SCRATCH_SUBDIR_TEMPLATE`. Stale per-iteration dirs older than N hours and orphaned (no live process) are reaped continuously, not on demand.
   - **Solution search approach and hints:** when a pattern repeats and is slow, pin a known location or cache; when a pattern risks an external rate-limit, cache the upstream once; disk pressure (any mount over a threshold) triggers an immediate janitor pass over the scratch root; never let docker-owned root-rw files leak — clean them through a busybox container, not host `rm`. Re-enter whenever a new noisy or slow pattern emerges, or the previous run leaked scratch dirs beyond budget.
   - **Reward:** iteration variance from non-measured factors approaches zero — zero ban incidents, predictable per-iteration wall-clock, locations re-used across iterations, zero disk-full crashes, bounded scratch-dir count per repo (a leak triggers investigation, not just cleanup).
   - **Attention mechanism:** host metrics and scratch-dir counts are the channel — a mount crossing its disk-pressure threshold, or a per-repo scratch-dir count over budget, signals D6 to run a janitor pass.
7. **Problem (vLLM spin-up):** stand up tool-capable OpenAI-compatible chat-completion endpoints callable from inside Docker, with authentication enforced.
   - **Value:** D2's agent backend and D10's compactor cannot run without a tool-capable model endpoint reachable from inside docker containers; this problem owns that contract surface.
   - **Contract and constraints:** all endpoints in active use serve models from the same generation family — never mix generations, since prompts calibrated against one drift on another. When two consumers have meaningfully different throughput/depth tradeoffs they get separate endpoints on separate accelerators so they don't contend for slots. Per-consumer model identity and call-time options (thinking mode, etc.) live in the consumer's own Contract.
   - **Solution search approach and hints:** arrive at a working endpoint set through a ralph loop over container, model, and reverse-proxy config; converge by what consumers can actually call, not what the endpoint exposes in isolation. Re-enter on any consumer reporting degraded service.
   - **Reward:** consumers report uninterrupted service.
   - **Attention mechanism:** consumer call results are the channel — a consumer logging a degraded or failed completion is the signal that pulls D7 to re-check that consumer's endpoint.
8. **Problem (agent runtime spin-up):** stand up the configured agent runtime as a credentialed service reachable from the work host, with conversation lifecycle, event-stream emission, and a context-management primitive configured.
   - **Value:** D2's recipe search needs an agent runtime that accepts a prompt, executes against a working tree, emits structured events the harness consumes, and survives the deepest stages without losing trajectories to context overflow; this problem owns that surface.
   - **Contract and constraints:** the runtime exposes a callable interface (in-process SDK, HTTP API, or MCP socket) reachable from the harness, with tool invocation, conversation state across iterations, and event streaming. It is configured with at least one context-management primitive — an automatic condenser that folds older events into a summary at a threshold, or delegated subagents with their own context windows — so single-call context budget does not scale with repo depth and overflow is structural, not catastrophic. Every exposed surface shares D7's credential discipline: no unauthenticated bind, no credentials in trajectory state or in scratch dirs reaped by D6. Sub-context activity is captured in the event stream (D10) so the operator can audit when context was folded. The runtime has network reach to D5's caching proxy (same docker network or equivalent route) so every Maven invocation resolves through the cache, not direct upstream — without this, wall-clock balloons and upstream hosts ban us. Runtime identity is pinned per attempt so D2's PASS rates are comparable across passes. The runtime accepts N concurrent isolated sessions (own conversation id, workspace, event-stream slice), N dictated by D9; sessions share the LLM endpoint and proxy but never a workspace.
   - **Solution search approach and hints:** prefer maintained frameworks over reinventing tool-call iteration, context pruning, or conversation state; choose a runtime by what D2 can actually drive, not by what it advertises. Network reach goes through the same reverse-proxy + auth surface as the vLLM endpoints; in-process SDK use is permitted but not exempt from that credential discipline. If the agent stalls or repeats actions on the same failure across iterations, the missing input is usually structural source detail that needs sub-context — not a different prompt. Re-enter on any D2 stage reporting an agent-runtime failure unrelated to prompt or repo state, whenever context overflow drops a trajectory, or whenever a stage cluster shows the agent stalling without obvious cause.
   - **Reward:** D2 reports uninterrupted agent execution; fraction of stages where context overflow caused trajectory loss approaches zero; zero credentialless surfaces.
   - **Attention mechanism:** the agent event stream is the channel — a trajectory dropped to context overflow, a stall repeating actions on one failure, or an unauthenticated surface, signals D8 to re-examine the runtime.
9. **Problem (runner saturation):** keep every host compute resource (CPU, memory, GPU compute, GPU memory, disk I/O, network) in a healthy utilisation band while any parent loop is making progress; the binding constraint shifts across the loop's phases and is re-detected, not assumed.
   - **Value:** an idle host wastes budget; any over-saturated resource drops the loop's wall-clock-per-PASS reward to noise. The binding constraint is rarely CPU alone — under N-way agent parallelism it shifts from CPU at startup to GPU compute during agent rounds to disk/network during mvn cache warming to GPU KV-cache memory under sustained batching.
   - **Contract and constraints:** composes with the parent loop rather than standing alone — saturation decisions are conditional on which loop is active, its recent action history, and which resource is nearest its band edge; any concurrency dial the agent can reach is in scope. The primary output is a per-loop concurrency recommendation N (read by D2 to size its per-pass session fan-out into D8's runtime); N may shift mid-pass as the binding constraint shifts.
   - **Solution search approach and hints:** sample every relevant resource periodically (host CPU + load avg, host memory, per-GPU utilisation + memory + KV-cache from the vLLM logs, disk I/O on the work mount, network to D5's proxy and D7's endpoint, docker daemon health); learn the band per resource; re-detect the binding constraint when the loop changes phase. GPU compute-percent snapshots are bursty under batching — read the average over a window, not the instant. Run continuously, dampened against oscillation; pause when no parent loop is active.
   - **Reward:** sustained healthy band across every relevant resource without thrashing or stalling the parent loop; the binding constraint correctly identified at any moment.
   - **Attention mechanism:** the resource samples are the channel — any resource (CPU, GPU compute/KV-cache, disk I/O, network, from `host_metrics` + the vLLM logs) nearing its band edge signals D9 to re-detect the binding constraint and re-issue N.
10. **Problem (compact observations):** capture system streams continuously to disk and run a compacting model over them to produce the orchestrator-visible digest.
   - **Value:** raw observation streams exceed the orchestrator's context budget; without a faithful digest, decisions are made from the wrong fraction of the data and trust in the digest erodes silently.
   - **Contract and constraints:** the compactor model is Qwen 3.6 27B AWQ (dense, served from D7), called with `chat_template_kwargs.enable_thinking: true`; fall back to no-thinking only when thinking output is unparseable. Capture runs regardless of whether anyone is reading it (raw streams keyed for retrieval); the compactor is never load-bearing for irreversible decisions without a spot check against the raw source. The digest is multi-resolution — at least four logarithmically spaced windows (currently `10s / 60s / 10m / 60m`) sampled side-by-side, so anomalies show by window-to-window contrast; gauges report first/last/min/max per window and counters report deltas, so adjacent windows differ on slow-moving signals instead of looking frozen. The captured set includes the agent-runtime event stream (D8, summary-only so the compactor's budget doesn't scale with investigator depth) alongside Vector's docker / app_logs / host_metrics. Any problem execution whose run window overlaps a gap in capture is invalid.
   - **Solution search approach and hints:** Vector is the canonical capture layer — it runs as the `obs-vector` container and sinks streams as JSONL to `/var/log/observe/` (`docker.jsonl`, `host_metrics.jsonl`, `app_logs.jsonl`); the compactor reads tails of those files rather than polling subprocesses; new streams enter by being added to `/tmp/vector.toml`, not by extending the compactor. The snapshot fed to the compactor must surface failure signals when present, not just happy-path state; emit only when state changes materially since the previous snapshot so a quiet stream stays quiet; on input exceeding budget, chunk-and-merge rather than drop; periodically spot-sample raw vs digest to recalibrate trust. Re-enter whenever a new noisy stream enters the loop.
   - **Reward:** balance between compaction rate and anomalies missed; quiet streams stay quiet so emitted entries carry signal.
   - **Attention mechanism:** the agent-runtime event stream is the channel that matters most — D10 watches it for a *change in agent behaviour* (a new error signature, a new bail label, an agent looping or over-verifying, a throughput stall), surfaced by multi-window contrast (10s / 60s / 5m / 30m) against the established baseline and emitted only when it did not occur before; steady state stays silent. Infra deltas (docker, host metrics) are secondary — if the agent event stream is not in the captured set, D10 is blind to the changes that matter.
