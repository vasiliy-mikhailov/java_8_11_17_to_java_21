# AGENTS.md

**Per-attempt history:** each `attempt_N/README.md` is a historical snapshot of the AGENTS.md state under which that attempt ran. It is not read by the agent — it exists only for audit.

0. **Fitness (writing this file):** keep AGENTS.md compact and outcome-named.
   - **Purpose:** compact outcome-named files force the agent to re-derive the *how* each iteration from its tools and the corpus; without this discipline the file rots as the domain shifts and the agent over-fits to one iteration's specifics.
   - **Contract and constraints:** A fitness function is written for an intelligent agent that searches for an implementation in a fuzzy environment, using the available tooling, resources, and harness in Ralph loops — so no implementation details should be supplied to the agent once it has demonstrated the ability to fill them in on its own. Fitness functions are read by humans many times, so they must be kept as short as possible. Every fitness in this file follows the same five-section structure as this one — Purpose, Contract and constraints, Search hints, Reward, Repeat. The Contract and constraints section is operator-only: the agent does not edit it; it is the immovable boundary within which the agent's other sections optimize. Section roles: Purpose names why the fitness exists (would the project notice if it were removed?); Contract and constraints names the operator-defined boundary; Search hints record what observed failures have taught the agent to try first; Reward names a count, fraction, or directly comparable artefact the operator can read off without judgement; Repeat names the trigger that brings the agent back to this fitness. The fitness one-liner names what the fitness produces, not how — mechanism belongs inside Contract and constraints. Backtick-pinned concretes (paths, command shapes, file names, concrete element names) are scaffolding the reader uses to recognize the situation the rule applies to and stay through trims; enumerations that age (lists of versions, models, repo names) get stripped. AGENTS.md is organized in clusters: meta (ff #0), the recipe and its falsifiability (ff #1–#2), and the substrate that lets the recipe loop run (ff #3–#10). Within a cluster, ffs are ordered by what serves what.
   - **Search hints:** read → why → intent — when revisiting a clause, ask "why is this here?"; strip back to the rule when the answer is mechanism the agent already fills itself, a bare plea for the rule, or an aged enumeration; keep when the answer scopes when the rule applies, names a concrete that lets the reader recognize the situation, or signals when the rule should be revisited.
   - **Reward:** cuts that lose words without losing the rule or its scope; every ff's Reward, after the pass, names something the operator can read off without judgement.
   - **Repeat:** every editing pass; additionally, whenever an ff's Contract and constraints is touched by the operator, the next pass over that ff re-audits its agent-mutable sections.

## Recipe cluster

1. **Fitness (recipe):** for every repo in `java21-migration-dataset.json`, the paste-into-any-coding-agent prompt executed by the configured agent runtime (ff #7) against the repo at `sha_from` produces a migrated tree that builds clean on `jv_to` and conserves the `jv_from` passing-test set under ff #10's criterion.
   - **Purpose:** an empirical migration recipe that beats the one-shot baseline is the primary research goal; the per-stage iterative loop is how the search converges across heterogeneous repos in the corpus.
   - **Contract and constraints:** the prompt is the deliverable artifact — self-contained for a stranger's repo, no references to our per-stage trajectory layout, no assumption of a specific backend or agent runtime; the same ff #0 discipline applies (outcome-named, no instructions the agent fills itself, search hints derived only from observed failure patterns). Writes per-repo trajectories under `attempt_N/per_repo_iter/<slug>/trajectory.json` (agent runtime + backend model + prompt fingerprint, event-stream summary, verdict, failure observation, wall-clock, migrated-tree diff vs `sha_from`); reads ff #3's dataset for the stage list; reads ff #5's host paths; consumes ff #4, #6, #7, #8, #9 transparently. Per-repo search is iterative — run the prompt under the configured agent runtime against the current working tree, on FAIL append the `[ERROR]` block + maven summary + the agent's own session summary into `observation_addendum.md` (per-stage operator-authored enrichment that rides on the in-harness invocation as the authoritative override for that stage, overriding generic search hints for the same failure cluster; for paste-into-agent use, the operator hand-edits the prompt instead), retry; a session whose action fingerprint matches one already tried bails. Agent runtime, backend model, and prompt version are pinned per attempt so PASS rate is comparable across passes. The corpus is processed as a multi-pass round-robin — each pass gives every still-FAILing repo K more attempts continuing its trajectory, then moves on; iterate passes until one yields zero new PASSes.
   - **Search hints:** craft the prompt the way AGENTS.md is crafted (see ff #0); never restate what the agent does well naturally; pair every hint with the observed failure mode that motivates it and prune the hint once that failure stops recurring; keep worked examples when trimming generic instruction — examples are scaffolding the agent imitates, and the abstract rule alone is weaker. When the agent applies the right step shape but consistently picks the wrong target, the gap is information-availability not hint quality — enrich `observation_addendum.md` with the specific pom element / classpath dump / source-grep result the agent needs, rather than enumerating possible targets in the prompt. Regime change: when the harness, recipe artifact set, or environment is altered in a way that materially changes what counts as a failure, INVALIDATE the trajectories of every stage whose history was conditioned on the old regime; cheapest invalidation is deleting the affected trajectory files so the next pass starts those stages from scratch. When the agent cannot reach a fix regardless of prompt or observation enrichment (requires a primitive that doesn't exist), ship that primitive deterministically and surface it in the prompt, not more hints. Per-step log dirs prevent successive same-named phases from overwriting each other; default K=1 — pass N grants budget=N attempts per still-FAILing stage, trading deep iteration for fast feedback and a natural checkpoint between passes to apply harness fixes, primitive additions, or prompt tweaks; trajectory state persists between passes so each new attempt picks up prior session summary.
   - **Reward:** corpus PASS rate vs the one-shot `UpgradeToJava<jv_to>` baseline; tie-breaker — fewer agent iterations to PASS; secondary signal — mean iterations-to-PASS across passing stages and reduction in iteration waste (dupe-session submissions, same-failure-signature repeats).
   - **Repeat:** continuous round-robin until convergence; restart from cached trajectories whenever the prompt changes materially. After per-repo converges, the universal-sequence question (one chain winning across many repos) is answered by clustering winning chains.
2. **Fitness (operator-runnable reproducer):** the project's current best-known deterministic conversion chain is always available as a single committed executable artifact a stranger runs on a Maven project from a vanilla machine, without any project-internal infrastructure.
   - **Purpose:** ff #1's PASS rate is a private number until a stranger can rerun it on their own machine and see the same number; this fitness owns the bridge from "works for us" to "works for anyone" and is the only thing that turns ff #1 into a scientific claim.
   - **Contract and constraints:** one script at a stable repo path, pushed before being considered shipped; `--help` is the only documentation the stranger needs; pinned to the recipe-artifact GAVs the latest attempt's runner empirically uses; embeds every harness escape that lives in the docker entry script (the standalone is a snapshot of the harness state, never a fork); a smoke pair of one canonical PASS stage and one canonical FAIL stage from the corpus is re-run before each commit and must keep its verdicts.
   - **Search hints:** regenerate when ff #1's PASS rate moves materially, when the docker entry script gains a new escape, when a pinned GAV bumps, or when a new deterministic primitive enters the chain; an unexpected smoke-pair verdict flip is a regression signal, not a green light.
   - **Reward:** smoke pair stays green; the stranger reproduces the project's deterministic best without any project knowledge beyond `--help`.
   - **Repeat:** every Search trigger.

## Substrate cluster

3. **Fitness (dataset rediscovery):** curate `java21-migration-dataset.json` as distinct-owner lineage samples balanced per (oldest-Java-version × dependency family) cell.
   - **Purpose:** a corpus biased toward already-prep'd states would let ff #1 hit high PASS rates without doing real migration work; the dataset must sample real migration windows so the recipe's value is measurable.
   - **Contract and constraints:** emits `java21-migration-dataset.json`; ff #1 reads it for the stage list and excludes any stage whose `sha_from` either fails `mvn compile` under jv_from or has zero passing tests (which would make ff #10's test-conservation criterion degenerate). Each entry is one (repo, jv_from→jv_to) stage; `sha_from` is picked by starting from `next_version_commit` (the first commit at the next Java version in the repo's lineage), walking back N commits along `--first-parent`, and accepting a commit that compiles AND whose unit-test suite passes a meaningful majority of tests under jv_from. `sha_to` is the version-bump commit at the target Java version. A full-lineage J8→J11→J17→J21 repo yields THREE stages (J8→J21, J11→J21, J17→J21), each using the same `sha_to` (the J21-bump) and a different commit-distance-back `sha_from` from the relevant intermediate-bump. Each entry records `pre_pass_count` (passing-test cardinality at `sha_from`) and `commits_back` (distance from `next_version_commit`) so ff #1 can weight stages.
   - **Search hints:** distinct-owner sampling; widen under-represented cells; prefer full-lineage repos over partial-lineage when both fit a cell; skip monorepos and giant clones that hang the walker; accept a candidate `sha_from` iff it compiles under jv_from AND passes a soft majority (default ≥0.7) of its non-skipped tests; probe at increasing commit-distances back (start 20, then 50, 100, 200) and accept the smallest distance meeting the bar; flag any stage whose `sha_from`→`sha_to` diff touches only the Java-version property or compiler-plugin source/target/release knob — the step-back wasn't deep enough and the stage is measuring property mutation, not recipe quality.
   - **Reward:** coverage in under-represented cells; corpus size at or above 200; fraction of entries with full J8→J11→J17→J21 lineage (each contributes 3 stages); fraction of entries where `sha_from` carries ≥1 passing test (so ff #10's criterion is enforceable, not skipped via empty-pre fallback); coverage of popular breaking-change boundaries (Spring Boot 2.x→3.x, javax→jakarta, Hibernate 5→6, JUnit 4→5) across the corpus — when full-lineage repos are scarce, fall back to single-step (jv_from → jv_from+1) stages that nevertheless exercise these breaking changes; a small corpus rich in framework-migration diversity is more valuable than a large corpus of trivial property-flip migrations.
   - **Repeat:** continuous; paused when ff #1 is saturated on the current corpus.
4. **Fitness (proxy dependency resolution):** every build's external-artifact resolution goes through a local caching proxy with plural upstream mirrors.
   - **Purpose:** without a caching proxy, build failures get attributed to upstream availability blips rather than code state, polluting the parent loop's reward signal.
   - **Contract and constraints:** the proxy caches every artifact it serves and survives across iterations. Consumers (ff #1, ff #3) attribute build failures to code state rather than upstream availability, so an unresolved artifact triggers a proxy widening before that build is counted toward the parent's reward.
   - **Search hints:** include both live mirrors and archival ones in the upstream set; container builds reach the proxy by container-network DNS, not host IPs; when a build fails on "cannot resolve X", widen the upstream set before treating the failure as code-attributable.
   - **Reward:** per-artifact cache-hit ratio; resolution failures distinguishable from compile failures in the parent loop's classification.
   - **Repeat:** whenever a parent loop's failures cluster on artifact resolution.
5. **Fitness (maintain local environment):** known stable paths, scratch-space discipline, and disk/cache stewardship for every other fitness function.
   - **Purpose:** iteration outcomes must vary only with what's being measured; unstable paths, shared scratch space, and resource contention drift the measurement noise floor up and obscure the signal; one ff exhausting a shared resource crashes another and corrupts the parent loop's results.
   - **Contract and constraints:** project workdir at `$HOME/java_8_11_17_to_java_21` (the one location pinned in this ff — every other path is an env var resolved from `.env` so the value lives next to credentials). Env vars consumers read: `MIRROR_CACHE_DIR` (per-repo git mirrors live as `<owner>/<repo>.git` under it); `SCRATCH_ROOTS` (colon-separated writable mounts ordered by preference; agent picks the first with sufficient headroom); `SCRATCH_SUBDIR_TEMPLATE` (relative template expanded per scratch dir, e.g. `tmp/<repo_name>/<ff_specific_subdir>/`). All build toolchains and recipe execution run inside Docker; SSH calls to the work host share one session, not one per command; free host resources held by unrelated work when this project needs them. Individual ff modules declare a scratch-dir name; this fitness resolves it against `SCRATCH_ROOTS` + `SCRATCH_SUBDIR_TEMPLATE`. Stale per-iteration dirs older than N hours and orphaned (no live process holding them) are reaped continuously, not on demand.
   - **Search hints:** when a pattern repeats and is slow, pin a known location or cache for it; when a pattern risks an external rate-limit, cache the upstream once; disk pressure (any mount over a threshold) triggers immediate janitor pass over the project's scratch root; never let docker-owned root-rw files leak — clean them through a busybox container, not host `rm`.
   - **Reward:** zero ban incidents; predictable per-iteration wall-clock; locations re-used across iterations; zero disk-full crashes during any parent loop; bounded scratch-dir count per repo (a leak indicator triggers investigation, not just cleanup).
   - **Repeat:** whenever a new noisy or slow pattern emerges, or whenever the previous run leaked scratch dirs beyond the bounded budget.
6. **Fitness (vLLM spin-up):** stand up tool-capable OpenAI-compatible chat-completion endpoints callable from inside Docker, with authentication enforced.
   - **Purpose:** ff #1's agent backend and ff #9's compactor cannot run without a tool-capable model endpoint reachable from inside docker containers; this fitness owns that contract surface.
   - **Contract and constraints:** all endpoints in active use serve models from the same generation family — never mix generations, since prompts calibrated against one generation drift on another. When two consumers have meaningfully different throughput / depth tradeoffs they get separate endpoints on separate accelerators so they don't contend for slots. Per-consumer model identity and call-time options (thinking mode, etc.) live in the consumer's own Contract.
   - **Search hints:** arrive at a working endpoint set through a ralph loop over container choice, model choice, and reverse-proxy config; converge by what the consumers can actually call, not what the endpoint exposes in isolation.
   - **Reward:** consumers report uninterrupted service.
   - **Repeat:** on any consumer reporting degraded service.
7. **Fitness (agent runtime spin-up):** stand up the configured agent runtime as a credentialed service reachable from the work host, with conversation lifecycle, event-stream emission, and context-management primitive configured.
   - **Purpose:** ff #1's recipe search cannot run without an agent runtime that accepts a prompt, executes against a working tree, emits structured events the harness can consume, and survives the deepest stages without losing trajectories to context overflow; this fitness owns that contract surface.
   - **Contract and constraints:** the runtime exposes a callable interface (in-process SDK, HTTP API, or MCP socket) reachable from the harness; supports tool invocation, conversation state across iterations, and event streaming. The runtime is configured with at least one context-management primitive — an automatic conversation condenser that folds older events into a summary at a configured threshold, or the ability to spawn delegated subagents with their own context windows — so the agent's single-call context budget does not scale with repo depth and overflow is structural, not catastrophic. Every exposed surface (server endpoint, conversation API, MCP socket, container exec interface) shares the credential discipline of ff #6's vLLM endpoints — no unauthenticated bind, no credentials in trajectory state, no credentials in scratch dirs reaped by ff #5. Sub-context activity is captured in the event stream (ff #9) so the operator can audit when and how context was folded. Runtime identity is pinned per attempt so ff #1's PASS rates are comparable across passes.
   - **Search hints:** prefer maintained frameworks over reinventing tool-call iteration, context pruning, or conversation state; choose a runtime by what ff #1 can actually drive, not by what the runtime advertises. When the runtime needs network reach (server mode, MCP), it goes through the same reverse-proxy + auth surface as the vLLM endpoints; in-process SDK use is permitted but does not exempt the runtime from the credential discipline of any external endpoint it dials. If the agent stalls or repeats actions on the same failure across iterations, the missing input is usually structural source detail that needs sub-context — not a different prompt.
   - **Reward:** ff #1 reports uninterrupted agent execution; fraction of stages where context overflow caused trajectory loss (target: zero); operator-spot-check rate at which post-condense trajectory still names the structural cause correctly; zero credentialless surfaces in agent-runtime configuration.
   - **Repeat:** on any ff #1 stage reporting an agent runtime failure unrelated to the prompt or repo state; whenever context overflow drops a trajectory; whenever a stage cluster shows the agent stalling without obvious cause.
8. **Fitness (runner saturation):** keep the verifier host CPU in a healthy utilisation band while any parent loop is making progress.
   - **Purpose:** an idle host while a parent loop has work waiting is wasted budget; an over-saturated host thrashes; either way the parent loop's wall-clock-per-PASS reward is lied to.
   - **Contract and constraints:** composes with the parent loop rather than standing alone — saturation decisions are made conditional on which loop is active and what its recent action history is; any concurrency dial the agent can reach is in scope.
   - **Search hints:** sample load periodically, decide what to adjust given the recent action history and which parent loop is active.
   - **Reward:** sustained band hit without thrashing or stalling the parent loop.
   - **Repeat:** continuous, dampened against oscillation; pause when no parent loop is active.
9. **Fitness (compact observations):** capture system streams continuously to disk and run a compacting model over them to produce the orchestrator-visible digest.
   - **Purpose:** raw observation streams exceed the orchestrator's context budget; without a faithful digest, decisions are made from the wrong fraction of the data and the operator's trust in the digest erodes silently.
   - **Contract and constraints:** the compactor model is Qwen 3.6 27B AWQ (dense, served from ff #6), called with `chat_template_kwargs.enable_thinking: true`; fall back to no-thinking only when thinking output is unparseable. Capture runs regardless of whether anyone is reading it (raw streams keyed for retrieval); the compactor is never load-bearing for irreversible decisions without a spot check against the raw source. The digest is multi-resolution — at least four logarithmically spaced windows (currently `10s / 60s / 10m / 60m`) are sampled side-by-side, so anomalies are spotted by window-to-window contrast rather than from a single cut; gauges report first/last/min/max per window and counters report deltas, so adjacent windows differ on slow-moving signals instead of looking frozen. The captured stream set includes the agent runtime event stream (ff #7, summary-only so the compactor's context budget does not scale with investigator depth) alongside Vector's docker / app_logs / host_metrics. Any fitness execution whose run window overlaps a gap in capture is invalid.
   - **Search hints:** Vector is the canonical capture layer; it already runs as the `obs-vector` container and sinks observed streams as JSONL to `/var/log/observe/` (`docker.jsonl`, `host_metrics.jsonl`, `app_logs.jsonl`); the compactor reads tails of those JSONL files rather than polling subprocesses; new observable streams enter the loop by being added to `/tmp/vector.toml`, not by extending the compactor. The snapshot fed to the compactor must surface failure signals when present, not just happy-path state; emit only when state changes materially since the previous snapshot, so a quiet stream stays quiet; on input exceeding the compactor's budget, chunk-and-merge rather than drop; periodically spot-sample raw vs digest to recalibrate trust.
   - **Reward:** balance between compaction rate and anomalies missed; quiet streams stay quiet so emitted entries carry signal.
   - **Repeat:** whenever a new noisy stream enters the loop.
10. **Fitness (test-conservation harness):** instrument ff #1's runner so the stronger PASS criterion ("every test passing pre-run must still pass post-run") is measurable.
    - **Purpose:** without this harness, ff #1's test-conservation clause is a promise rather than an enforcement, and compile-only PASS over-rewards runs that quietly break tests — corpus PASS-rate becomes meaningless.
    - **Contract and constraints:** at the start of every agent-driven run, execute `mvn test` against the unmodified clone under jv_from; parse the per-test outcomes from `target/surefire-reports/TEST-*.xml`; capture the set `pre_pass = {(test class, test method)}` that passed (skipped/failed tests don't enter `pre_pass`). At the end of the run, execute `mvn test` against the post-run tree under jv_to; capture `post_pass` the same way. The run FAILs iff `pre_pass ⊄ post_pass` — i.e. any tuple in `pre_pass` is missing from `post_pass`. Empty `pre_pass` (project has no tests, or all were already broken) falls back to compile-only PASS so we don't penalise corpus repos that never had test coverage.
    - **Search hints:** parse surefire XML directly off the host-bound work dir after each test-phase docker run; clear `target/surefire-reports/` before each run so pre and post don't share state; cap each `mvn test` at a fixed wall-clock so a single flaky/long-running test class doesn't stall the loop; persist the regressed-test list into the trajectory so the agent sees which specific tests broke and the operator can spot-check.
    - **Reward:** fraction of stages where the runner produced a valid pre_pass + post_pass measurement (vs aborted on test infrastructure); fraction of "compile-only PASS" stages that flip to FAIL under test conservation (signal: how much the compile-only criterion was over-rewarding).
    - **Repeat:** whenever ff #1's prompt or recipe set changes materially.
