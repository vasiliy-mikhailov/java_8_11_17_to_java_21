# AGENTS.md

**Per-attempt history:** each `attempt_N/README.md` is a historical snapshot of the AGENTS.md state under which that attempt ran. It is not read by the agent — it exists only for audit.

0. **Fitness (writing this file):** keep AGENTS.md compact and outcome-named so the agent re-derives the *how* every iteration from its tools and the corpus.
   - **Why:** compact outcome-named files force the agent to re-derive the how each iteration; without this discipline the file rots as the domain shifts and the agent over-fits to one iteration's specifics.
   - **Contract and constraints:** every fitness in this file follows the same five-section structure as this one — Why, Contract (or Contract and constraints), Search hints, Reward, Repeat. Constraints on writing any of those sections: no implementation instructions the agent can fill itself; no enumerations that age; no justifications that merely plead the case for the rule. Justifications that scope WHEN the rule applies, or signal HOW to recognize the rule should be reconsidered, ARE kept — without them the rule becomes opaque and gets misapplied.
   - **Search hints:** read → why → intent — when revisiting a clause, ask "why is this here?"; if the answer is implementation detail, dated enumeration, or a bare plea for the rule, strip it back to the rule itself; if the answer scopes when the rule applies or signals when it should be revisited, keep it.
   - **Reward:** cuts that lose words without losing the rule or its scope.
   - **Repeat:** every editing pass.
1. **Fitness (recipe):** for every repo in `java21-migration-dataset.json`, find an OpenRewrite recipe SEQUENCE of `(label, jdk, recipes)` steps that builds clean on jv_to under `mvn compile` at every step, not just the last, AND conserves the test result set — every (test class, test method) tuple that PASSED under `mvn test` against the unmodified source on jv_from must still PASS under `mvn test` against the post-recipe source on jv_to; tests that were already failing before the recipe don't count against the recipe, but breaking a previously-passing test is a FAIL. The baseline to beat per repo is the one-shot `UpgradeToJava<jv_to>` recipe. Per-repo search is iterative — run the current chain, on failure feed the proposer the `[ERROR]` block + maven summary and the prior-attempt history, take its mutated chain, retry. A chain identical to one already tried bails. Declarative YAML recipes only. Each step's pom/source edits persist into the next step's working tree. The corpus is processed as a multi-pass round-robin — each pass gives every still-FAILing repo K more attempts continuing its trajectory, then moves on; iterate passes until one yields zero new PASSes. The universal-sequence question (one chain winning across many repos) is answered after per-repo converges, by clustering winning chains.
   - **Why:** an empirical migration recipe that beats the one-shot baseline is the primary research goal; the per-stage iterative loop is how the search converges across heterogeneous repos in the corpus.
   - **Contract:** writes per-repo trajectories under `attempt_N/per_repo_iter/<slug>/trajectory.json` (chain attempted, verdict, failure observation, proposer rationale, wall-clock); reads item 2's dataset for the stage list; reads item 4's host paths; consumes items 3, 5, 6 transparently.
   - **Search hints:** per-step log dirs prevent successive same-named phases from overwriting each other; default K=1 — pass N grants budget=N attempts per still-FAILing stage, trading deep iteration for fast feedback and a natural checkpoint between passes to apply harness fixes, primitive additions, or prompt tweaks; trajectory state persists between passes so each new attempt picks up prior hypotheses.

   - **Reward:** corpus PASS rate vs the one-shot baseline; tie-breaker — smaller diff from baseline chain.
   - **Repeat:** continuous round-robin until convergence; restart from cached trajectories whenever the proposer changes materially.
2. **Fitness (dataset rediscovery):** curate `java21-migration-dataset.json` as distinct-owner lineage samples balanced per (oldest-Java-version × dependency family) cell. Each entry is one (repo, jv_from→jv_to) stage where the recorded `sha_from` is several commits BEFORE the version-bump commit — specifically: starting from `next_version_commit` (the first commit at the next Java version in the repo's lineage), walk back N commits along `--first-parent` and pick a commit that compiles AND whose unit-test suite passes a meaningful majority of tests under jv_from. The "N commits back" steps the search past the maintainer's pre-migration prep work, so the recipe has actual migration work to do rather than just mutating a property the maintainer had already prepared for. `sha_to` is the version-bump commit at the target Java version. A full-lineage J8→J11→J17→J21 repo therefore yields THREE stages (J8→J21, J11→J21, J17→J21), each using the same `sha_to` (the J21-bump) and a different commit-distance-back `sha_from` from the relevant intermediate-bump. Target corpus size ≥200 stages; preferring repos with the full lineage so item 1 can compare leap-vs-stepped on the same project.
   - **Why:** a corpus biased toward already-prep'd states would let item 1 hit high PASS rates without doing real migration work; the dataset must sample real migration windows so the recipe's value is measurable.
   - **Contract:** emits `java21-migration-dataset.json`; item 1 reads it for the stage list and excludes any stage whose `sha_from` either fails `mvn compile` under jv_from or has zero passing tests (which would make item 8's test-conservation criterion degenerate). Each entry records `pre_pass_count` (passing-test cardinality at `sha_from`) and `commits_back` (distance from `next_version_commit`) so item 1 can weight stages.
   - **Search hints:** distinct-owner sampling; widen under-represented cells; prefer full-lineage repos over partial-lineage when both fit a cell; skip monorepos and giant clones that hang the walker; for each candidate `sha_from`, run `mvn compile` then `mvn test` inside Docker under jv_from; accept the candidate iff compile is clean AND the pass-rate (passing / (passing + failing), ignoring skipped) is at least a soft majority (default ≥0.7); probe at increasing commit-distances back (start 20, then 50, 100, 200) — accept the smallest distance that meets the bar (closest to but clearly before the bump); flag any stage whose `sha_from`→`sha_to` diff touches only the Java-version property or compiler-plugin source/target/release knob — it indicates the step-back wasn't deep enough and the stage is measuring property mutation, not recipe quality; commit-distance (not calendar days) is the right unit because Java-era windows vary hugely in duration across repos.
   - **Reward:** coverage in under-represented cells; corpus size at or above 200; fraction of entries with full J8→J11→J17→J21 lineage (each contributes 3 stages); fraction of entries where `sha_from` carries ≥1 passing test (so item 8's criterion is enforceable, not skipped via empty-pre fallback); coverage of popular breaking-change boundaries (Spring Boot 2.x→3.x, javax→jakarta, Hibernate 5→6, JUnit 4→5) across the corpus — when full-lineage repos are scarce, fall back to single-step (jv_from → jv_from+1) stages that nevertheless exercise these breaking changes; a small corpus rich in framework-migration diversity is more valuable than a large corpus of trivial property-flip migrations.
   - **Repeat:** continuous; paused when item 1 is saturated on the current corpus.
3. **Fitness (proxy dependency resolution):** every build's external-artifact resolution goes through a local caching proxy with plural upstream mirrors. The proxy caches every artifact it serves and survives across iterations.
   - **Why:** without a caching proxy, build failures get attributed to upstream availability blips rather than code state, polluting the parent loop's reward signal.
   - **Contract:** consumers (items 1, 2) attribute build failures to code state rather than upstream availability, so an unresolved artifact triggers a proxy widening before that build is counted toward the parent's reward.
   - **Search hints:** include both live mirrors and archival ones in the upstream set; container builds reach the proxy by container-network DNS, not host IPs; when a build fails on "cannot resolve X", widen the upstream set before treating the failure as code-attributable.
   - **Reward:** per-artifact cache-hit ratio; resolution failures distinguishable from compile failures in the parent loop's classification.
   - **Repeat:** whenever a parent loop's failures cluster on artifact resolution.
4. **Fitness (maintain local environment):** known stable paths, scratch-space discipline, and disk/cache stewardship for every other fitness function — iteration outcomes do not drift for reasons unrelated to what's being measured, and one ff does not crash another by exhausting a shared resource.
   - **Why:** iteration outcomes must vary only with what's being measured; unstable paths, shared scratch space, and resource contention drift the measurement noise floor up and obscure the signal.
   - **Contract:** project workdir at `$HOME/java_8_11_17_to_java_21`; corpus repo-mirror cache at `/var/cache/git-mirrors/<owner>/<repo>.git`; all build toolchains and recipe execution run inside Docker; SSH calls to the work host share one session, not one per command; free host resources held by unrelated work when this project needs them. Every other ff's per-iteration scratch lives under `/mnt/fire/tmp/<repo_name>/<ff_specific_subdir>/` (`/mnt/fire` chosen because it has the most headroom; falls back to `/mnt/steam` then `/tmp` when fire is unavailable). Individual ff modules declare a scratch-dir name; ff #4 provisions the actual path. Stale per-iteration dirs older than N hours and orphaned (no live process holding them) are reaped continuously, not on demand.
   - **Search hints:** when a pattern repeats and is slow, pin a known location or cache for it; when a pattern risks an external rate-limit, cache the upstream once; disk pressure (any mount over a threshold) triggers immediate janitor pass over the project's scratch root; never let docker-owned root-rw files leak — clean them through a busybox container, not host `rm`.
   - **Reward:** zero ban incidents; predictable per-iteration wall-clock; locations re-used across iterations; zero disk-full crashes during any parent loop; bounded scratch-dir count per repo (a leak indicator triggers investigation, not just cleanup).
   - **Repeat:** whenever a new noisy or slow pattern emerges, or whenever the previous run leaked scratch dirs beyond the bounded budget.
5. **Fitness (vLLM spin-up):** stand up one or more OpenAI-compatible chat-completion endpoints that the agent can call from inside Docker containers, serving tool-capable models, with authentication enforced. Arrive there through a ralph loop over container, model, and reverse-proxy config.
   - **Why:** item 1's proposer and item 7's compactor cannot run without a tool-capable model endpoint reachable from inside docker containers; this fitness owns that contract surface.
   - **Contract:** consumers (item 1's proposer, item 7's compactor) default to thinking mode (`chat_template_kwargs.enable_thinking: true`); fall back to no-thinking only when thinking output is unparseable. All endpoints in active use serve models from the same generation family — never mix generations, since prompts calibrated against one generation drift on another. When two consumers have meaningfully different throughput / depth tradeoffs they get separate endpoints on separate accelerators so they don't contend for slots: the proposer (deep, long-context, intermittent) on a dense endpoint on the highest-VRAM accelerator; the compactor (high-frequency, structured-output, short) on a MoE endpoint on a smaller accelerator.
   - **Reward:** consumers report uninterrupted service.
   - **Repeat:** on any consumer reporting degraded service.
6. **Fitness (runner saturation):** keep the verifier host CPU in a healthy utilisation band while any parent loop is making progress — composes with the parent rather than standing alone.
   - **Why:** an idle host while a parent loop has work waiting is wasted budget; an over-saturated host thrashes; either way the parent loop's wall-clock-per-PASS reward is lied to.
   - **Contract:** any concurrency dial the agent can reach.
   - **Search hints:** sample load periodically, decide what to adjust given the recent action history and which parent loop is active.
   - **Reward:** sustained band hit without thrashing or stalling the parent loop.
   - **Repeat:** continuous, dampened against oscillation; pause when no parent loop is active.
7. **Fitness (compact observations):** capture system streams continuously to disk, raw and keyed for retrieval, regardless of whether anyone is reading them; a compacting model processes those streams to produce the orchestrator-visible digest.
   - **Why:** raw observation streams exceed the orchestrator's context budget; without a faithful digest, decisions are made from the wrong fraction of the data and the operator's trust in the digest erodes silently.
   - **Contract:** the compactor is never load-bearing for irreversible decisions without a spot check against the raw source; any fitness execution whose run window overlaps a gap in capture is invalid.
   - **Search hints:** Vector is the canonical capture layer; it already runs as the `obs-vector` container and sinks every observed stream as JSONL to `/var/log/observe/` (`docker.jsonl` from `docker_logs` source, `host_metrics.jsonl` from `host_metrics` source, `app_logs.jsonl` from a `file` source over `/tmp/*.log`); the compactor reads tails of those JSONL files, not subprocess polling of `docker ps`/`mpstat`/`ls`; new observable streams enter the loop by being added to `/tmp/vector.toml`, not by extending the compactor. The snapshot fed to the compactor must surface failure signals when present, not just happy-path state; emit only when state changes materially since the previous snapshot, so a quiet stream stays quiet; on input exceeding the compactor's budget, chunk-and-merge rather than drop; periodically spot-sample raw vs digest to recalibrate trust.
   - **Reward:** balance between compaction rate and anomalies missed; quiet streams stay quiet so emitted entries carry signal.
   - **Repeat:** whenever a new noisy stream enters the loop.
8. **Fitness (test-conservation harness):** instrument item 1's runner so the stronger PASS criterion ("every test passing pre-recipe must still pass post-recipe") is measurable. Without this harness, item 1's test-conservation clause is a promise, not an enforcement.
   - **Why:** compile-only PASS over-rewards chains that quietly break tests; without this harness, item 1's reward signal lies and the corpus PASS-rate number becomes meaningless.
   - **Contract:** at the start of every recipe-chain run, execute `mvn test` against the unmodified clone under jv_from; parse the per-test outcomes from `target/surefire-reports/TEST-*.xml`; capture the set `pre_pass = {(test class, test method)}` that passed (skipped/failed tests don't enter `pre_pass`). At the end of the chain, execute `mvn test` against the post-recipe tree under jv_to; capture `post_pass` the same way. The chain FAILs iff `pre_pass ⊄ post_pass` — i.e. any tuple in `pre_pass` is missing from `post_pass`. Empty `pre_pass` (project has no tests, or all were already broken) falls back to compile-only PASS so we don't penalise corpus repos that never had test coverage.
   - **Search hints:** parse surefire XML directly off the host-bound work dir after each test-phase docker run; clear `target/surefire-reports/` before each run so pre and post don't share state; cap each `mvn test` at a fixed wall-clock so a single flaky/long-running test class doesn't stall the loop; persist the regressed-test list into the trajectory so the proposer sees which specific tests broke and the operator can spot-check.
   - **Reward:** fraction of stages where the runner produced a valid pre_pass + post_pass measurement (vs aborted on test infrastructure); fraction of "compile-only PASS" stages that flip to FAIL under test conservation (signal: how much the compile-only criterion was over-rewarding).
   - **Repeat:** whenever item 1's chain shape or recipe set changes materially.
9. **Fitness (operator-runnable reproducer):** the project's current best-known deterministic conversion chain is always available as a single committed executable artifact a stranger runs on a Maven project from a vanilla machine, without any project-internal infrastructure.
   - **Why:** the project's best-known chain has to be runnable by a stranger on their own machine; if it's only runnable from inside our infrastructure, it isn't really shipped.
   - **Contract:** one script at a stable repo path, pushed before being considered shipped; `--help` is the only documentation the stranger needs; pinned to the recipe-artifact GAVs the latest attempt's runner empirically uses; embeds every harness escape that lives in the docker entry script (the standalone is a snapshot of the harness state, never a fork); a smoke pair of one canonical PASS stage and one canonical FAIL stage from the corpus is re-run before each commit and must keep its verdicts.
   - **Search hints:** regenerate when item 1's PASS rate moves materially, when the docker entry script gains a new escape, when a pinned GAV bumps, or when a new deterministic primitive enters the chain; an unexpected smoke-pair verdict flip is a regression signal, not a green light.
   - **Reward:** smoke pair stays green; the stranger reproduces the project's deterministic best without any project knowledge beyond `--help`.
   - **Repeat:** every Search trigger.
10. **Fitness (proposer cultivation):** the LLM proposer reaches PASS in as few attempts as possible on the stages it can solve at all, and clearly converges or bails on the stages it can't.
   - **Why:** the proposer is the bottleneck on the recipe loop's compute cost — every extra attempt is wall-clock and tokens; cultivating its prompt is the cheapest reward-per-effort lever in the system.
   - **Contract:** the proposer's prompt is its fitness function; the same discipline ff #0 applies to AGENTS.md applies here — outcome-named, no instructions the proposer already executes itself, search hints derived only from observed failure patterns. The proposer reads its prompt, the failure observation from the latest chain, and a compacted history of prior attempts on this stage, and emits exactly one chain proposal.
   - **Search hints:** craft the prompt the way AGENTS.md is crafted (see ff #0); never restate what the proposer does well naturally; pair every hint with the observed failure mode that motivates it and prune the hint once that failure stops recurring; keep worked examples when trimming generic instruction — examples are scaffolding the proposer imitates, and the abstract rule alone is weaker. Chain-shape hints: prefer stepped framework upgrades over one-shot leaps; pure JDK upgrades exempt from staging. When the proposer applies the right step shape but consistently picks the wrong target, the gap is information-availability not hint quality — enrich the observation with the specific pom element / classpath dump / source-grep result the proposer needs, rather than enumerating possible targets in the prompt. Pre-recipe pom-XML edit: when `mvn rewrite:run` fails because the unmodified project itself doesn't compile (JDK-removed javax classes still imported, or annotation processors pinned at versions too old for the target JDK), no chain mutation can recover — the fix is a direct pom XML edit ahead of any rewrite step, including elements like `<annotationProcessorPaths>` that OpenRewrite primitives don't reach. Regime change: when the harness, recipe artifact set, or environment is altered in a way that materially changes what counts as a failure, INVALIDATE the trajectories of every stage whose history was conditioned on the old regime; cheapest invalidation is deleting the affected trajectory files so the next pass starts those stages from scratch. When the proposer cannot reach a fix regardless of prompt or observation enrichment (requires a primitive that doesn't exist), ship that primitive deterministically in the harness, not more hints.
   - **Reward:** mean attempts-to-PASS across passing stages; reduction in attempt-waste (dupe-chain submissions, same-failure-signature repeats); fraction of clusters that converge on the first hint-aware attempt vs require iteration.
   - **Repeat:** whenever a new failure cluster emerges, whenever observed proposer waste on a cluster exceeds half of attempts spent, or when a hint stops earning its place (the failure it addressed no longer recurs).
