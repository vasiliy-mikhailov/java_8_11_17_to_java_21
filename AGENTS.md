# AGENTS.md

**Per-attempt history:** each `attempt_N/README.md` snapshots the AGENTS.md state that attempt ran under — audit only, not read by the agent.

1. **Problem (writing this file):** keep AGENTS.md compact and outcome-named.
   - **Value:** compact, outcome-named files force the agent to re-derive the *how* each iteration from its tools and the corpus; without this discipline the file rots as the domain shifts and the agent over-fits to one iteration's specifics.
   - **Contract and constraints:** A problem is a self-amplifying attractor — one autonomous concern with a single extremum (its Reward) and a single trigger (its Attention mechanism) that pulls the agent's attention and amplifies until satisfied. Each is a concern the operator has offloaded to the agent (the agent watches, triggers, and drives it), making this file a delegation protocol, not a checklist. Only one problem is in foreground at a time (Ukhtomsky); before every action the agent verifies exactly one is dominant — concurrent problems signal confusion to resolve first. Control passes when a stronger trigger fires or the current extremum is approached; an interrupted problem is resumed, not restarted, so each persists enough state (trajectory files, addenda, in-progress search position) for resumption to be genuine. Problems are written for an intelligent agent searching a fuzzy environment with the available tooling in Ralph loops: supply no implementation detail the agent can fill itself, and — since humans reread this file often — keep each as short as possible. Every problem has five sections: **Value** (why it exists — would the project notice its removal?); **Contract and constraints** (the operator-only boundary the other sections optimize within — the agent does not edit it); **Solution search approach and hints** (what observed failures taught the agent to try first, plus this agent's own loop or trigger for returning to the problem); **Reward** (one extremum the operator reads off without judgement — a count, fraction, or comparable artefact); **Attention mechanism** (how a change this problem makes or notices in the shared environment becomes salient to whichever problem must react — the channel it publishes a signal to or watches, e.g. a log line, a metric, a `feedback.jsonl` entry, or an observability stream — as distinct from the agent's own re-entry loop). The one-liner names what the problem produces, not how. Backtick-pinned concretes (paths, command shapes, file and element names) are recognition scaffolding and survive trims; aging enumerations (versions, models, repo names) get stripped. The file is clustered — meta (P1); recipe and setup (P2–P4); substrate (P5–P10); portability witness (P11) — and within a cluster ordered by what serves what. The open attempt is always the directory `current_attempt`; every tool references `current_attempt/...` (never a hardcoded number). An attempt gets its number only at archival: closing it renames `current_attempt` → `attempt_N` and leaves it frozen in place — so opening the next attempt needs no scripts, just that rename (no symlink to move). Attempt-agnostic shared helpers live in a frozen attempt (e.g. `attempt_7/tools`) only while never modified; the moment the open attempt must edit one — e.g. the build/test entry script `run_one_stage_v2.sh` or the `docker_phase`/`shallow_fetch` harness it reaches through — it is copied under `current_attempt/` and referenced sibling-relative (resolve siblings via `__file__`, never a hardcoded `attempt_N` path), so the tuned harness ships and is versioned with the attempt while frozen attempts stay byte-frozen.
   - **Solution search approach and hints:** read → why → intent — for each clause ask "why is this here?". Strip when the answer is mechanism the agent already fills, a bare plea for the rule, or an aged enumeration; keep when it scopes when the rule applies, names a recognition concrete, or signals when to revisit. A topic that always fires inside another problem's loop is mechanism of that loop — it belongs in that loop's Contract, not as its own problem. Re-run every editing pass; additionally, whenever the operator touches a problem's Contract and constraints, the next pass re-audits that problem's agent-mutable sections.
   - **Reward:** cuts that lose words without losing the rule or its scope; after the pass, every problem's Reward names something the operator can read off without judgement.
   - **Attention mechanism:** the file is the channel — an operator edit to a problem's Contract, or a problem's agent-mutable sections visibly bloating, is the signal that pulls this meta-problem to re-audit that problem.

## Recipe cluster

2. **Problem (Java LTS bump artifact for any coding agent):** ship one `bump-java-version` skill — a portable skill package consumable by any tool-using coding agent (no assumption of our harness, scratch layout, or runtime) — that on each next-LTS bump (8→11, 11→17, 17→21) drives the highest corpus PASS rate.
   - **Value:** an empirical migration skill that beats the one-shot OpenRewrite baseline is the primary research goal; portability across agents is the audience constraint, so the deliverable is self-contained and harness-agnostic.
   - **Contract and constraints:** the deliverable is the `bump-java-version` skill — a self-contained, portable skill package: a `SKILL.md` in the six-section form P3 pins, the recipe catalog as versioned-FQN dependencies, and the per-LTS `bump_<from>_to_<to>.sh` scripts; loadable by any tool-using coding agent that supports skills (no assumption of our harness, scratch layout, or runtime), all stage-agnostic (stage facts arrive via the consumer's harness as a prepended Stage header). Reads P4 for the stage list. Per-stage validation is delegated to P3, which returns `{terminal_rung, terminal_outcome, artifact_updates}`; fold `artifact_updates` into the corpus `bump-java-version` skill iff that does not regress corpus reward, else leave it unchanged and record the stage unintegrated. Concurrency across stages from P9.
   - **Solution search approach and hints:** when a failure family's stages only pass at the strongest rung (Opus) and won't harden down to OH+Qwen, that family lacks a corpus-level remedy — commission a recipe via P3's search or add a failure-output row. An `artifact_updates` that passes its own stage but regresses others on rollout is over-fit — generalise its trigger or scope the fix narrower. Loop continuously round-robin until reward stops improving, restarting from cached trajectories whenever the corpus artifact's sha materially changes.
   - **Reward:** corpus PASS rate (fraction of stages P3 terminates at the OH+Qwen rung with `terminal_outcome=PASS`) vs the one-shot `UpgradeToJava<jv_to>` baseline.
   - **Attention mechanism:** `feedback.jsonl` is the channel — a failure family clustering there, or its stages only passing at the strong end of the rung ladder (stuck at Opus), signals that the family needs a corpus-level remedy.
3. **Problem (per-repo skill + recipe escalation):** for each (repo, commit) stage P2 hands down, harden the current best (skill, recipes) artifact until it drives the bump to PASS or labelled BAIL under OpenHands+Qwen — by making the stage pass on a ladder of executors, all driven by Claude — routinely two rungs: first Claude+Opus 4.8, then Claude managing OpenHands+Qwen (the production target), with Claude patching the artifact at the stronger rung so it survives the weaker one. A third executor, Claude+bare-Qwen (the same Qwen as OH+Qwen but in a minimal harness), is NOT a routine rung but an on-demand probe (see Value).
   - **Value:** P2 sees corpus PASS/FAIL rates but cannot distinguish "stage genuinely unfixable", "production LLM too weak", "production harness wrong", and "artifact has a gap". This problem separates those for one stage: the routine rungs name most of the failure mode — fails even at Opus → genuinely unfixable or needs a new recipe; reaches Opus but not OH+Qwen → the skill is either unclear to a weak LLM or incompatible with OH's tool-use conventions. The on-demand bare-Qwen probe resolves that last ambiguity (passes bare-Qwen but fails OH+Qwen → an OH-harness limit for P8; fails bare-Qwen too → the skill needs hardening) and, run on a sample, witnesses the skill's portability (P2's no-assumption-of-our-harness: it must pass in a second, minimal harness, not only inside OpenHands) — so P2 acts on signal, not noise. It is also the path by which a single stage's skill/recipe insight propagates into the corpus artifact.
   - **Contract and constraints:** The (skill, recipes) artifact has a pinned shape so every rung reads the same structure and a failure maps to a known section. The skill's `SKILL.md` is one file whose six sections — Purpose, Contract and constraints, Basic flow, Failure outputs → remedies, Reward, When to bail — are pinned in this order (the live instance is the `bump-java-version` skill at `.agents/skills/bump-java-version/`, whose `SKILL.md` carries the six sections alongside bundled `scripts/` and `references/`). Two pins a reader needs here: the Failure-outputs table's fix vocabulary is {pom edit, `mvn rewrite:run` by FQN, docker cleanup, labelled bail, ignore-with-justification}, and the Reward (PASS) is `mvn compile` succeeds under jv_to AND pre-pass test set ⊆ post-pass test set. Recipes are whatever the table names by FQN; they live in the cache pool the bump scripts pull from, with identities (not source) pinned per artifact revision. Stage facts (repo URL, sha, jv_from, jv_to, workdir) are injected by each rung's harness as a prepended Stage header, never in the `SKILL.md`, so the skill stays portable. The search is one schema: per stage, the **search results** — the artifact version run at each rung (`opus`, then `oh_qwen`; `qwen` only when the probe is invoked) with its outcome — and the one **best search result**, the hardened artifact that reaches the OH+Qwen rung. Claude is the sole searcher at every rung: it runs the current artifact under that rung's executor and, on failure, patches the artifact (rewrite any `SKILL.md` section, add or reorder failure rows, add recipe FQNs publishing to the catalog, edit bump scripts); the executor (Opus 4.8, Qwen 27B FP8, or OpenHands+Qwen 27B FP8) only runs what Claude ships. The routine ladder goes Opus → OH+Qwen (strong → production): Opus surfaces "genuinely unfixable / needs a new recipe"; OH+Qwen is the production target the artifact must reach; a fix made at Opus is re-validated at OH+Qwen, since it can regress. The bare-Qwen rung is off the critical path — invoked only (a) on a periodic sample to confirm the skill still passes in a minimal harness (portability), and (b) to assign an ambiguous OH+Qwen failure to either the skill or the OH harness; it never gates a stage's terminal outcome. A rung's FAIL is re-run-confirmed (flakes re-run at the same rung, not escalated); the stage terminates when the OH+Qwen rung reaches PASS or labelled BAIL. Per (stage × rung × validation-run) the harness appends one line to `current_attempt/feedback.jsonl` (`rung ∈ {opus, qwen, oh_qwen}`, `outcome ∈ {PASS, BAIL:<label>, FAIL}`) and, before invoking a rung, snapshots the exact `SKILL.md` to `skill_snapshots/<sha>.md` and catalog to `recipe_snapshots/<sha>.jar` (content-addressed, idempotent); the full per-stage record is `per_repo_iter/<slug>/trajectory.json` alongside `oh_dialogue.log` and `bump.diff`. Each rung's session is isolated (own workdir, conversation id, surefire path); concurrency bounded by P9. Output to P2: `{stage, terminal_rung, terminal_outcome, artifact_returned}`, where `terminal_rung` is the weakest rung reached at PASS or labelled BAIL (ideally `oh_qwen`) and `artifact_returned` is the hardened artifact; P2 decides folding.
   - **Solution search approach and hints:** fast-fail at a rung — if Claude+executor stalls (no progress for >N actions on the same step) treat it as FAIL at that rung, don't spin; never recurse on flakes (one-off network error, Docker glitch) — re-run the same rung. On a FAIL, Claude's first probe is to read the executor's actual trace (`oh_dialogue.log`) to find where its reasoning diverged from the skill — typically a missed trigger paraphrase, a skipped step, or a tool-use convention the `SKILL.md` didn't anticipate. Claude's patch is the smallest possible (a one-sentence tightening, a row reword, a step reorder), not a wholesale rewrite; add a new recipe only when no prose change could fix the executor's behaviour. After patching at Opus, re-validate at OH+Qwen, since a fix can regress. A stage that keeps failing at the OH+Qwen rung even after patching is either an OH structural limit (surface to P8, not a skill rewrite) or a genuinely unclear skill — invoke the bare-Qwen probe to tell which: passes there → P8; fails there → harden the skill. Run the probe on a periodic sample regardless, to catch portability drift (the skill coming to depend on OH scaffolding). Re-invoke per stage each pass from P2, and again when the corpus artifact's sha changes and the stage previously did not reach PASS at the OH+Qwen rung.
   - **Reward:** fraction of input stages where the OH+Qwen rung reaches PASS or labelled BAIL — both initially and after each patch; the metric is "OH+Qwen-rung success rate", not "any-rung success rate". Secondary: average patches per stage to reach OH+Qwen success (lower is better).
   - **Attention mechanism:** `feedback.jsonl` is the channel — a re-run-confirmed FAIL trajectory written there is the signal that pulls Claude to patch the artifact at that rung; an OH+Qwen rung still failing after patches signals P8 (a structural harness limit), not another skill rewrite.
4. **Problem (dataset = repo list + per-run randomized-baseline sampler):** maintain `dataset-repos.json` (repo names only); draw the eval set fresh each round by sampling, per repo, a random commit that declares Java 8/11/17 AND compiles under that JDK.
   - **Value:** a fixed-sha corpus lets P2 (and its GEPA/EvoSkills optimizers) overfit to specific commits, and biases toward already-prep'd states that score high without real migration work. Sampling a *different* valid baseline per run makes the eval a moving target — only improvements that generalize across each repo's history survive. It also retires baseline curation: validity is checked when a baseline is drawn, not maintained as a clean-list.
   - **Contract and constraints:** the persistent dataset is `dataset-repos.json` (a list of `owner/repo`). Each round, `tools/sample_shas.py --seed <round>` emits `dataset-shas.json` (committed, lives in the attempt folder) — **N = 100 datapoints/attempt** for now, a seeded random subset of `dataset-repos.json` (the count rises as we move toward GEPA) — of entries `{repo, sha, jv_from, jv_to}` where `jv_from ∈ {8,11,17}`, `jv_to` is the next LTS (8→11, 11→17, 17→21), and the `sha` is **verified to test-compile under `jv_from` at sample time**. P2 reads the per-run file. A different `--seed` ⇒ different shas; the same seed is reproducible. A repo that yields no compiling 8/11/17 baseline within the attempt bound contributes nothing that round — never a false `baseline-broken` into P2. Junk (already ≥ `jv_to`, no pom, non-compiling) cannot enter the dataset because the sampler rejects it.
   - **Solution search approach and hints:** bounded rejection search per repo — shuffle commits with the run seed, walk them, cheap-reject (no pom / jv ∉ {8,11,17}) for free, and `test-compile` each eligible candidate under its `jv_from`; accept the first that compiles, capped at ~10 compile attempts (plus a scan cap on commits inspected). `test-compile` (main+test) is the validity bar so P2 never inherits a baseline whose tests don't even compile. Sampling across **full** history organically surfaces 8→11 and 11→17 hops, not just 17→21 — fetch unshallowed so the walk sees real ancestry. Keep `dataset-repos.json` broad (distinct owners, dependency-family spread); for a repo that rarely yields a baseline, raise `--max-attempts` rather than hand-curating. Use a fresh seed per optimization round and score edits on the **mean across seeds**, discarding seed-specific gains.
   - **Reward:** sampler hit-rate (fraction of `dataset-repos.json` that yields a valid compiling 8/11/17 baseline per run) and the LTS-hop diversity of the sampled set; and, for the optimizer, **low variance of P2's score across seeds** — a genuinely general artifact scores about the same on any sampled baseline set, an overfit one does not.
   - **Attention mechanism:** the per-run `dataset-shas.json` and the spread of P2's score across seeds are the channel — a repo that never yields a baseline (raise its attempt bound or drop it from `dataset-repos.json`), or a P2 gain that evaporates under a new seed (an overfit edit — reject it), is the signal that pulls P4 to adjust sampling.

## Substrate cluster

5. **Problem (proxy dependency resolution):** every build's external-artifact resolution goes through a local caching proxy with plural upstream mirrors.
   - **Value:** without a caching proxy, build failures get attributed to upstream availability blips rather than code state, polluting the parent loop's reward signal.
   - **Contract and constraints:** the proxy caches every artifact it serves and survives across iterations. Consumers (P2, P4) attribute build failures to code state, so an unresolved artifact triggers a proxy widening before that build counts toward reward.
   - **Solution search approach and hints:** include both live and archival mirrors; container builds reach the proxy by container-network DNS, not host IPs; when a build fails on "cannot resolve X", widen the upstream set before treating the failure as code-attributable. Re-enter whenever a parent loop's failures cluster on artifact resolution.
   - **Reward:** per-artifact cache-hit ratio (higher is better; resolution failures become distinguishable from compile failures in the parent loop's classification).
   - **Attention mechanism:** build logs are the channel — a `cannot resolve X` line is the signal that pulls P5 to widen the upstream set before that failure is counted as code-attributable.
6. **Problem (maintain local environment):** known stable paths, scratch-space discipline, and disk/cache stewardship for every other problem.
   - **Value:** iteration outcomes must vary only with what's being measured; unstable paths, shared scratch, and contention drift the noise floor up, and one problem exhausting a shared resource crashes another and corrupts the parent loop's results.
   - **Contract and constraints:** project workdir at `$HOME/java_8_11_17_to_java_21` (the one path pinned here — every other is an env var resolved from `.env` so the value lives next to credentials). Env vars consumers read: `MIRROR_CACHE_DIR` (per-repo git mirrors as `<owner>/<repo>.git`); `SCRATCH_ROOTS` (colon-separated writable mounts by preference; agent picks the first with headroom); `SCRATCH_SUBDIR_TEMPLATE` (relative template per scratch dir, e.g. `tmp/<repo_name>/<ff_specific_subdir>/`). All build toolchains and recipe execution run inside Docker; SSH calls to the work host share one session, not one per command; free host resources held by unrelated work when this project needs them. **Docker container logs are rotated** — the daemon's default `json-file` driver has no rotation, so an unbounded container log (one reached 55 GB here) silently fills the disk and crashes dependent services (e.g. P10's compactor on its next write); `observe_rotate.py` truncates over-cap container json-logs via a root container, and the permanent fix is docker daemon `log-opts` (`max-size`/`max-file`) applied on a maintenance restart. A problem declares a scratch-dir name; this problem resolves it against `SCRATCH_ROOTS` + `SCRATCH_SUBDIR_TEMPLATE`. Each run reaps its own scratch workdir on completion (in a `finally`); because the build container writes root-owned files, the reap runs through a root container, not host `rm`. A scratch dir that outlives its run is a reap failure to investigate.
   - **Solution search approach and hints:** when a pattern repeats and is slow, pin a known location or cache; when a pattern risks an external rate-limit, cache the upstream once; never let docker-owned root-rw files leak — each run cleans its own scratch through a root container (not host `rm`) in a `finally`, so cleanup is never skipped or deferred. Re-enter whenever a new noisy or slow pattern emerges, or when a run's scratch survives the run — that means the per-run reap failed, so fix the reap.
   - **Reward:** iteration variance from non-measured factors approaches zero — zero ban incidents, predictable per-iteration wall-clock, locations re-used across iterations, zero disk-full crashes, bounded scratch-dir count per repo (a leak triggers investigation, not just cleanup).
   - **Attention mechanism:** host metrics and scratch-dir counts are the channel — a scratch dir that outlives its run, or a per-repo scratch-dir count creeping up, signals that the per-run reap failed and P6 must fix it.
7. **Problem (vLLM spin-up):** stand up tool-capable OpenAI-compatible chat-completion endpoints callable from inside Docker, with authentication enforced.
   - **Value:** P2's agent backend and P10's compactor cannot run without a tool-capable model endpoint reachable from inside docker containers; this problem owns that contract surface.
   - **Contract and constraints:** all endpoints in active use serve models from the same generation family — never mix generations, since prompts calibrated against one drift on another. When two consumers have meaningfully different throughput/depth tradeoffs they get separate endpoints on separate accelerators so they don't contend for slots. A single public gateway (`inference.mikhailov.tech`, a Caddy reverse proxy) fronts both endpoints — `/qwen-3.6-27b-fp8/v1` reaches the FP8 proposer (GPU0) and `/qwen-3.6-27b-awq/v1` the AWQ observability compactor (GPU1) — the path names the served model; one hostname and credential, so consumers never hardcode internal container IPs (which churn on daemon restart), and unmatched paths return a 404. Per-consumer model identity and call-time options (thinking mode, etc.) live in the consumer's own Contract.
   - **Solution search approach and hints:** arrive at a working endpoint set through a ralph loop over container, model, and reverse-proxy config; converge by what consumers can actually call, not what the endpoint exposes in isolation. Re-enter on any consumer reporting degraded service.
   - **Reward:** consumers report uninterrupted service.
   - **Attention mechanism:** consumer call results are the channel — a consumer logging a degraded or failed completion is the signal that pulls P7 to re-check that consumer's endpoint.
8. **Problem (agent runtime spin-up):** stand up the configured agent runtime as a credentialed service reachable from the work host, with conversation lifecycle, event-stream emission, and a context-management primitive configured.
   - **Value:** P2's recipe search needs an agent runtime that accepts a prompt, executes against a working tree, emits structured events the harness consumes, and survives the deepest stages without losing trajectories to context overflow; this problem owns that surface.
   - **Contract and constraints:** the runtime exposes a callable interface (in-process SDK, HTTP API, or MCP socket) reachable from the harness, with tool invocation, conversation state across iterations, and event streaming. It is configured with at least one context-management primitive — an automatic condenser that folds older events into a summary at a threshold, or delegated subagents with their own context windows — so single-call context budget does not scale with repo depth and overflow is structural, not catastrophic. Every exposed surface shares P7's credential discipline: no unauthenticated bind, no credentials in trajectory state or in scratch dirs reaped by P6. Sub-context activity is captured in the event stream (P10) so the operator can audit when context was folded. The runtime has network reach to P5's caching proxy (same docker network or equivalent route) so every Maven invocation resolves through the cache, not direct upstream — without this, wall-clock balloons and upstream hosts ban us. Runtime identity is pinned per attempt so P2's PASS rates are comparable across passes. The runtime accepts N concurrent isolated sessions (own conversation id, workspace, event-stream slice), N dictated by P9; sessions share the LLM endpoint and proxy but never a workspace.
   - **Solution search approach and hints:** prefer maintained frameworks over reinventing tool-call iteration, context pruning, or conversation state; choose a runtime by what P2 can actually drive, not by what it advertises. Network reach goes through the same reverse-proxy + auth surface as the vLLM endpoints; in-process SDK use is permitted but not exempt from that credential discipline. If the agent stalls or repeats actions on the same failure across iterations, the missing input is usually structural source detail that needs sub-context — not a different prompt. Re-enter on any P2 stage reporting an agent-runtime failure unrelated to prompt or repo state, whenever context overflow drops a trajectory, or whenever a stage cluster shows the agent stalling without obvious cause.
   - **Reward:** P2 reports uninterrupted agent execution; fraction of stages where context overflow caused trajectory loss approaches zero; zero credentialless surfaces.
   - **Attention mechanism:** the agent event stream is the channel — a trajectory dropped to context overflow, a stall repeating actions on one failure, or an unauthenticated surface, signals P8 to re-examine the runtime.
9. **Problem (runner saturation):** keep every host compute resource (CPU, memory, GPU compute, GPU memory, disk I/O, network) in a healthy utilisation band while any parent loop is making progress; the binding constraint shifts across the loop's phases and is re-detected, not assumed.
   - **Value:** an idle host wastes budget; any over-saturated resource drops the loop's wall-clock-per-PASS reward to noise. The binding constraint is rarely CPU alone — under N-way agent parallelism it shifts from CPU at startup to GPU compute during agent rounds to disk/network during mvn cache warming to GPU KV-cache memory under sustained batching.
   - **Contract and constraints:** composes with the parent loop rather than standing alone — saturation decisions are conditional on which loop is active, its recent action history, and which resource is nearest its band edge; any concurrency dial the agent can reach is in scope. The primary output is a per-loop concurrency recommendation N (read by P2 to size its per-pass session fan-out into P8's runtime); N may shift mid-pass as the binding constraint shifts.
   - **Solution search approach and hints:** sample every relevant resource periodically (host CPU + load avg, host memory, per-GPU utilisation + memory + KV-cache from the vLLM logs, disk I/O on the work mount, network to P5's proxy and P7's endpoint, docker daemon health); learn the band per resource; re-detect the binding constraint when the loop changes phase. GPU compute-percent snapshots are bursty under batching — read the average over a window, not the instant. Run continuously, dampened against oscillation; pause when no parent loop is active.
   - **Reward:** sustained healthy band across every relevant resource without thrashing or stalling the parent loop; the binding constraint correctly identified at any moment.
   - **Attention mechanism:** the resource samples are the channel — any resource (CPU, GPU compute/KV-cache, disk I/O, network, from `host_metrics` + the vLLM logs) nearing its band edge signals P9 to re-detect the binding constraint and re-issue N.
10. **Problem (compact observations):** capture system streams continuously to disk and run a compacting model over them to produce the orchestrator-visible digest.
   - **Value:** raw observation streams exceed the orchestrator's context budget; without a faithful digest, decisions are made from the wrong fraction of the data and trust in the digest erodes silently.
   - **Contract and constraints:** the compactor model is Qwen 3.6 27B AWQ (dense, served from P7 via the gateway's `/qwen-3.6-27b-awq/v1` route, GPU1), called with `chat_template_kwargs.enable_thinking: false`, and its context budget is matched to that endpoint's `max-model-len` (64K; oversized inputs are chunked across calls, not truncated). Capture runs regardless of whether anyone is reading it (raw streams keyed for retrieval). The Vector capture sinks under `/var/log/observe/*.jsonl` are **size-rotated** (`observe_rotate.py` under a keepalive, truncate-in-place over a cap; root-owned sinks via a root container) so an unbounded sink cannot fill the disk and crash the compactor on its next digest write — disk-full is how the digest goes silently stale; the compactor itself runs under a keepalive that auto-restarts it. (Rotation of the Docker container logs that feed these sinks lives in P6.) The compactor is never load-bearing for irreversible decisions without a spot check against the raw source. The digest is multi-resolution — at least four logarithmically spaced windows (currently `10s / 60s / 10m / 60m`) sampled side-by-side, so anomalies show by window-to-window contrast; gauges report first/last/min/max per window and counters report deltas, so adjacent windows differ on slow-moving signals instead of looking frozen. The captured set includes the agent-runtime event stream (P8, summary-only so the compactor's budget doesn't scale with investigator depth) alongside Vector's docker / app_logs / host_metrics. Any problem execution whose run window overlaps a gap in capture is invalid.
   - **Solution search approach and hints:** Vector is the canonical capture layer — it runs as the `obs-vector` container and sinks streams as JSONL to `/var/log/observe/` (`docker.jsonl`, `host_metrics.jsonl`, `app_logs.jsonl`); the compactor reads tails of those files rather than polling subprocesses; new streams enter by being added to `/tmp/vector.toml`, not by extending the compactor. The snapshot fed to the compactor must surface failure signals when present, not just happy-path state; emit only when state changes materially since the previous snapshot so a quiet stream stays quiet; on input exceeding budget, chunk-and-merge rather than drop; periodically spot-sample raw vs digest to recalibrate trust. Re-enter whenever a new noisy stream enters the loop.
   - **Reward:** balance between compaction rate and anomalies missed; quiet streams stay quiet so emitted entries carry signal.
   - **Attention mechanism:** the agent-runtime event stream is the channel that matters most — P10 watches it for a *change in agent behaviour* (a new error signature, a new bail label, an agent looping or over-verifying, a throughput stall), surfaced by multi-window contrast (10s / 60s / 5m / 30m) against the established baseline and emitted only when it did not occur before; steady state stays silent. Infra deltas (docker, host metrics) are secondary — if the agent event stream is not in the captured set, P10 is blind to the changes that matter.

## Portability witness

11. **Problem (portability witness — the skill runs under a stranger's agent on a from-scratch host):** prove `bump-java-version` drives a green next-LTS bump when an off-the-shelf coding agent (e.g. `opencode`) consumes it in a container built fresh with only the two JDKs (`jv_from`, `jv_to`) + git + network — no `~/bin/mvn` wrapper, no Nexus, no warm `.m2`, no OpenHands.
   - **Value:** P2 demands the skill assume nothing about our harness, yet every P3 rung runs it INSIDE that harness (wrapper JDK-selection, Nexus mirror, pre-warmed cache, our skill-loader), so a skill can score high and still be silently un-portable — welded to `JDK=<n> mvn`, to a private recipe artifact, to a warm cache. This is the only witness with none of those: it converts "passes here" into "a stranger can run it", and each failure names exactly one coupling to sever before the artifact ships.
   - **Contract and constraints:** the container is built from a public base, given exactly two JDKs at known homes plus the project's own `./mvnw` (NO system Maven — "plain `jv_from`/`jv_to` access" is the test), resolving from Maven Central directly (cold `.m2`, no mirror). The driver is an off-the-shelf tool-using agent pointed at the P7 OpenAI-compatible endpoint, handed the skill as plain instructions + scripts (not via our skill-loader). The skill's only inputs are the workdir and the two JDK homes; whatever it needs beyond a stock JDK + `mvnw` + Central it must carry itself — the OpenRewrite recipe artifact resolves from a public repo (`tech.mikhailov.bump_java_version_recipes` published to Maven Central) or the skill bundles and `install:install-file`s it. The witnessed repo's baseline must be green IN THIS CONTAINER; a baseline that needs external infra (a DB, our cache) is disqualified from the witness, not counted a skill failure.
   - **Solution search approach and hints:** couplings observed first — scripts hard-wired `JDK=<n> mvn` (our wrapper's env-JDK selection; real Maven needs `JAVA_HOME`), a private `tech.mikhailov…` recipe artifact only in Nexus, reliance on a warm `.m2`. Sever them: scripts set `JAVA_HOME` and prefer `./mvnw`; the recipe artifact goes to a public repo (or is bundled + installed); pick a repo whose tests are self-contained (no DB/integration infra). Re-enter whenever the skill's scripts, its recipe dependencies, or the driving agent change.
   - **Reward:** of the witnessed repos, the fraction reaching a green `jv_to` build with tests conserved on a from-scratch container — plus the list of severed couplings.
   - **Attention mechanism:** the fresh-container build + agent-run log is the channel — an artifact-not-found-on-Central, a `mvn: command not found`, a wrong-JDK compile, or a baseline that isn't green in-container each names the next coupling to sever, and feeds back to P2/P3.
