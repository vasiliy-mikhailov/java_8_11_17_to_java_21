# AGENTS.md

**Per-attempt history:** each `attempt_N/README.md` is a historical snapshot of the AGENTS.md state under which that attempt ran. It is not read by the agent — it exists only for audit.

0. **Fitness (writing this file):** keep AGENTS.md compact and outcome-named.
   - **Purpose:** compact outcome-named files force the agent to re-derive the *how* each iteration from its tools and the corpus; without this discipline the file rots as the domain shifts and the agent over-fits to one iteration's specifics.
   - **Contract and constraints:** A fitness function is written for an intelligent agent that searches for an implementation in a fuzzy environment, using the available tooling, resources, and harness in Ralph loops — so no implementation details should be supplied to the agent once it has demonstrated the ability to fill them in on its own. Fitness functions are read by humans many times, so they must be kept as short as possible. Every fitness in this file follows the same five-section structure as this one — Purpose, Contract and constraints, Search hints, Reward, Repeat. The Contract and constraints section is operator-only: the agent does not edit it; it is the immovable boundary within which the agent's other sections optimize. Section roles: Purpose names why the fitness exists (would the project notice if it were removed?); Contract and constraints names the operator-defined boundary; Search hints record what observed failures have taught the agent to try first; Reward names a count, fraction, or directly comparable artefact the operator can read off without judgement; Repeat names the trigger that brings the agent back to this fitness. The fitness one-liner names what the fitness produces, not how — mechanism belongs inside Contract and constraints. Backtick-pinned concretes (paths, command shapes, file names, concrete element names) are scaffolding the reader uses to recognize the situation the rule applies to and stay through trims; enumerations that age (lists of versions, models, repo names) get stripped.
   - **Search hints:** read → why → intent — when revisiting a clause, ask "why is this here?"; strip back to the rule when the answer is mechanism the agent already fills itself, a bare plea for the rule, or an aged enumeration; keep when the answer scopes when the rule applies, names a concrete that lets the reader recognize the situation, or signals when the rule should be revisited.
   - **Reward:** cuts that lose words without losing the rule or its scope; every ff's Reward, after the pass, names something the operator can read off without judgement.
   - **Repeat:** every editing pass; additionally, whenever an ff's Contract and constraints is touched by the operator, the next pass over that ff re-audits its agent-mutable sections.
1. **Fitness (recipe):** for every repo in `java21-migration-dataset.json`, find an OpenRewrite recipe SEQUENCE of `(label, jdk, recipes)` steps that builds clean on jv_to at every step (not just the last) and conserves the jv_from passing-test set under item 8's criterion.
   - **Purpose:** an empirical migration recipe that beats the one-shot baseline is the primary research goal; the per-stage iterative loop is how the search converges across heterogeneous repos in the corpus.
   - **Contract and constraints:** writes per-repo trajectories under `attempt_N/per_repo_iter/<slug>/trajectory.json` (chain attempted, verdict, failure observation, proposer rationale, wall-clock); reads item 2's dataset for the stage list; reads item 4's host paths; consumes items 3, 5, 6 transparently. Per-repo search is iterative — run the current chain, on failure feed the proposer the `[ERROR]` block + maven summary + prior-attempt history, take its mutated chain, retry; a chain identical to one already tried bails. Declarative YAML recipes only. Each step's pom/source edits persist into the next step's working tree. The corpus is processed as a multi-pass round-robin — each pass gives every still-FAILing repo K more attempts continuing its trajectory, then moves on; iterate passes until one yields zero new PASSes.
   - **Search hints:** per-step log dirs prevent successive same-named phases from overwriting each other; default K=1 — pass N grants budget=N attempts per still-FAILing stage, trading deep iteration for fast feedback and a natural checkpoint between passes to apply harness fixes, primitive additions, or prompt tweaks; trajectory state persists between passes so each new attempt picks up prior hypotheses.
   - **Reward:** corpus PASS rate vs the one-shot `UpgradeToJava<jv_to>` baseline; tie-breaker — smaller diff from baseline chain.
   - **Repeat:** continuous round-robin until convergence; restart from cached trajectories whenever the proposer changes materially. After per-repo converges, the universal-sequence question (one chain winning across many repos) is answered by clustering winning chains.
2. **Fitness (dataset rediscovery):** curate `java21-migration-dataset.json` as distinct-owner lineage samples balanced per (oldest-Java-version × dependency family) cell.
   - **Purpose:** a corpus biased toward already-prep'd states would let item 1 hit high PASS rates without doing real migration work; the dataset must sample real migration windows so the recipe's value is measurable.
   - **Contract and constraints:** emits `java21-migration-dataset.json`; item 1 reads it for the stage list and excludes any stage whose `sha_from` either fails `mvn compile` under jv_from or has zero passing tests (which would make item 8's test-conservation criterion degenerate). Each entry is one (repo, jv_from→jv_to) stage; `sha_from` is picked by starting from `next_version_commit` (the first commit at the next Java version in the repo's lineage), walking back N commits along `--first-parent`, and accepting a commit that compiles AND whose unit-test suite passes a meaningful majority of tests under jv_from. `sha_to` is the version-bump commit at the target Java version. A full-lineage J8→J11→J17→J21 repo yields THREE stages (J8→J21, J11→J21, J17→J21), each using the same `sha_to` (the J21-bump) and a different commit-distance-back `sha_from` from the relevant intermediate-bump. Each entry records `pre_pass_count` (passing-test cardinality at `sha_from`) and `commits_back` (distance from `next_version_commit`) so item 1 can weight stages.
   - **Search hints:** distinct-owner sampling; widen under-represented cells; prefer full-lineage repos over partial-lineage when both fit a cell; skip monorepos and giant clones that hang the walker; accept a candidate `sha_from` iff it compiles under jv_from AND passes a soft majority (default ≥0.7) of its non-skipped tests; probe at increasing commit-distances back (start 20, then 50, 100, 200) and accept the smallest distance meeting the bar; flag any stage whose `sha_from`→`sha_to` diff touches only the Java-version property or compiler-plugin source/target/release knob — the step-back wasn't deep enough and the stage is measuring property mutation, not recipe quality.
   - **Reward:** coverage in under-represented cells; corpus size at or above 200; fraction of entries with full J8→J11→J17→J21 lineage (each contributes 3 stages); fraction of entries where `sha_from` carries ≥1 passing test (so item 8's criterion is enforceable, not skipped via empty-pre fallback); coverage of popular breaking-change boundaries (Spring Boot 2.x→3.x, javax→jakarta, Hibernate 5→6, JUnit 4→5) across the corpus — when full-lineage repos are scarce, fall back to single-step (jv_from → jv_from+1) stages that nevertheless exercise these breaking changes; a small corpus rich in framework-migration diversity is more valuable than a large corpus of trivial property-flip migrations.
   - **Repeat:** continuous; paused when item 1 is saturated on the current corpus.
3. **Fitness (proxy dependency resolution):** every build's external-artifact resolution goes through a local caching proxy with plural upstream mirrors.
   - **Purpose:** without a caching proxy, build failures get attributed to upstream availability blips rather than code state, polluting the parent loop's reward signal.
   - **Contract and constraints:** the proxy caches every artifact it serves and survives across iterations. Consumers (items 1, 2) attribute build failures to code state rather than upstream availability, so an unresolved artifact triggers a proxy widening before that build is counted toward the parent's reward.
   - **Search hints:** include both live mirrors and archival ones in the upstream set; container builds reach the proxy by container-network DNS, not host IPs; when a build fails on "cannot resolve X", widen the upstream set before treating the failure as code-attributable.
   - **Reward:** per-artifact cache-hit ratio; resolution failures distinguishable from compile failures in the parent loop's classification.
   - **Repeat:** whenever a parent loop's failures cluster on artifact resolution.
4. **Fitness (maintain local environment):** known stable paths, scratch-space discipline, and disk/cache stewardship for every other fitness function.
   - **Purpose:** iteration outcomes must vary only with what's being measured; unstable paths, shared scratch space, and resource contention drift the measurement noise floor up and obscure the signal; one ff exhausting a shared resource crashes another and corrupts the parent loop's results.
   - **Contract and constraints:** project workdir at `$HOME/java_8_11_17_to_java_21`; corpus repo-mirror cache at `/var/cache/git-mirrors/<owner>/<repo>.git`; all build toolchains and recipe execution run inside Docker; SSH calls to the work host share one session, not one per command; free host resources held by unrelated work when this project needs them. Every other ff's per-iteration scratch lives under `/mnt/fire/tmp/<repo_name>/<ff_specific_subdir>/` (`/mnt/fire` chosen because it has the most headroom; falls back to `/mnt/steam` then `/tmp` when fire is unavailable). Individual ff modules declare a scratch-dir name; ff #4 provisions the actual path. Stale per-iteration dirs older than N hours and orphaned (no live process holding them) are reaped continuously, not on demand.
   - **Search hints:** when a pattern repeats and is slow, pin a known location or cache for it; when a pattern risks an external rate-limit, cache the upstream once; disk pressure (any mount over a threshold) triggers immediate janitor pass over the project's scratch root; never let docker-owned root-rw files leak — clean them through a busybox container, not host `rm`.
   - **Reward:** zero ban incidents; predictable per-iteration wall-clock; locations re-used across iterations; zero disk-full crashes during any parent loop; bounded scratch-dir count per repo (a leak indicator triggers investigation, not just cleanup).
   - **Repeat:** whenever a new noisy or slow pattern emerges, or whenever the previous run leaked scratch dirs beyond the bounded budget.
5. **Fitness (vLLM spin-up):** stand up tool-capable OpenAI-compatible chat-completion endpoints callable from inside Docker, with authentication enforced.
   - **Purpose:** item 1's proposer and item 7's compactor cannot run without a tool-capable model endpoint reachable from inside docker containers; this fitness owns that contract surface.
   - **Contract and constraints:** all endpoints in active use serve models from the same generation family — never mix generations, since prompts calibrated against one generation drift on another. When two consumers have meaningfully different throughput / depth tradeoffs they get separate endpoints on separate accelerators so they don't contend for slots. Per-consumer model identity and call-time options (thinking mode, etc.) live in the consumer's own Contract.
   - **Search hints:** arrive at a working endpoint set through a ralph loop over container choice, model choice, and reverse-proxy config; converge by what the consumers can actually call, not what the endpoint exposes in isolation.
   - **Reward:** consumers report uninterrupted service.
   - **Repeat:** on any consumer reporting degraded service.
6. **Fitness (runner saturation):** keep the verifier host CPU in a healthy utilisation band while any parent loop is making progress.
   - **Purpose:** an idle host while a parent loop has work waiting is wasted budget; an over-saturated host thrashes; either way the parent loop's wall-clock-per-PASS reward is lied to.
   - **Contract and constraints:** composes with the parent loop rather than standing alone — saturation decisions are made conditional on which loop is active and what its recent action history is; any concurrency dial the agent can reach is in scope.
   - **Search hints:** sample load periodically, decide what to adjust given the recent action history and which parent loop is active.
   - **Reward:** sustained band hit without thrashing or stalling the parent loop.
   - **Repeat:** continuous, dampened against oscillation; pause when no parent loop is active.
7. **Fitness (compact observations):** capture system streams continuously to disk and run a compacting model over them to produce the orchestrator-visible digest.
   - **Purpose:** raw observation streams exceed the orchestrator's context budget; without a faithful digest, decisions are made from the wrong fraction of the data and the operator's trust in the digest erodes silently.
   - **Contract and constraints:** the compactor model is Qwen 3.6 27B AWQ (dense, served from item 5), called with `chat_template_kwargs.enable_thinking: true`; fall back to no-thinking only when thinking output is unparseable. Capture runs regardless of whether anyone is reading it (raw streams keyed for retrieval); the compactor is never load-bearing for irreversible decisions without a spot check against the raw source; any fitness execution whose run window overlaps a gap in capture is invalid.
   - **Search hints:** Vector is the canonical capture layer; it already runs as the `obs-vector` container and sinks observed streams as JSONL to `/var/log/observe/` (`docker.jsonl`, `host_metrics.jsonl`, `app_logs.jsonl`); the compactor reads tails of those JSONL files rather than polling subprocesses; new observable streams enter the loop by being added to `/tmp/vector.toml`, not by extending the compactor. The snapshot fed to the compactor must surface failure signals when present, not just happy-path state; emit only when state changes materially since the previous snapshot, so a quiet stream stays quiet; on input exceeding the compactor's budget, chunk-and-merge rather than drop; periodically spot-sample raw vs digest to recalibrate trust.
   - **Reward:** balance between compaction rate and anomalies missed; quiet streams stay quiet so emitted entries carry signal.
   - **Repeat:** whenever a new noisy stream enters the loop.
8. **Fitness (test-conservation harness):** instrument item 1's runner so the stronger PASS criterion ("every test passing pre-recipe must still pass post-recipe") is measurable.
   - **Purpose:** without this harness, item 1's test-conservation clause is a promise rather than an enforcement, and compile-only PASS over-rewards chains that quietly break tests — corpus PASS-rate becomes meaningless.
   - **Contract and constraints:** at the start of every recipe-chain run, execute `mvn test` against the unmodified clone under jv_from; parse the per-test outcomes from `target/surefire-reports/TEST-*.xml`; capture the set `pre_pass = {(test class, test method)}` that passed (skipped/failed tests don't enter `pre_pass`). At the end of the chain, execute `mvn test` against the post-recipe tree under jv_to; capture `post_pass` the same way. The chain FAILs iff `pre_pass ⊄ post_pass` — i.e. any tuple in `pre_pass` is missing from `post_pass`. Empty `pre_pass` (project has no tests, or all were already broken) falls back to compile-only PASS so we don't penalise corpus repos that never had test coverage.
   - **Search hints:** parse surefire XML directly off the host-bound work dir after each test-phase docker run; clear `target/surefire-reports/` before each run so pre and post don't share state; cap each `mvn test` at a fixed wall-clock so a single flaky/long-running test class doesn't stall the loop; persist the regressed-test list into the trajectory so the proposer sees which specific tests broke and the operator can spot-check.
   - **Reward:** fraction of stages where the runner produced a valid pre_pass + post_pass measurement (vs aborted on test infrastructure); fraction of "compile-only PASS" stages that flip to FAIL under test conservation (signal: how much the compile-only criterion was over-rewarding).
   - **Repeat:** whenever item 1's chain shape or recipe set changes materially.
9. **Fitness (operator-runnable reproducer):** the project's current best-known deterministic conversion chain is always available as a single committed executable artifact a stranger runs on a Maven project from a vanilla machine, without any project-internal infrastructure.
   - **Purpose:** the project's best-known chain has to be runnable by a stranger on their own machine; if it's only runnable from inside our infrastructure, it isn't really shipped.
   - **Contract and constraints:** one script at a stable repo path, pushed before being considered shipped; `--help` is the only documentation the stranger needs; pinned to the recipe-artifact GAVs the latest attempt's runner empirically uses; embeds every harness escape that lives in the docker entry script (the standalone is a snapshot of the harness state, never a fork); a smoke pair of one canonical PASS stage and one canonical FAIL stage from the corpus is re-run before each commit and must keep its verdicts.
   - **Search hints:** regenerate when item 1's PASS rate moves materially, when the docker entry script gains a new escape, when a pinned GAV bumps, or when a new deterministic primitive enters the chain; an unexpected smoke-pair verdict flip is a regression signal, not a green light.
   - **Reward:** smoke pair stays green; the stranger reproduces the project's deterministic best without any project knowledge beyond `--help`.
   - **Repeat:** every Search trigger.
10. **Fitness (proposer cultivation):** the LLM proposer reaches PASS in as few attempts as possible on the stages it can solve at all, and clearly converges or bails on the stages it can't.
   - **Purpose:** the proposer is the bottleneck on the recipe loop's compute cost — every extra attempt is wall-clock and tokens; cultivating its prompt is the cheapest reward-per-effort lever in the system.
   - **Contract and constraints:** the proposer model is Qwen 3.6 27B FP8 (dense, served from item 5), called with `chat_template_kwargs.enable_thinking: true`; fall back to no-thinking only when thinking output is unparseable. The proposer's prompt is its fitness function; the same discipline ff #0 applies to AGENTS.md applies here — outcome-named, no instructions the proposer already executes itself, search hints derived only from observed failure patterns. The proposer reads its prompt, the failure observation from the latest chain, and a compacted history of prior attempts on this stage, and emits exactly one chain proposal. Per-stage operator-authored prompt enrichment, if present at `attempt_N/per_repo_iter/<slug>/observation_addendum.md`, rides on the proposer call as the authoritative override for that stage (overrides generic search hints for the same failure cluster).
   - **Search hints:** craft the prompt the way AGENTS.md is crafted (see ff #0); never restate what the proposer does well naturally; pair every hint with the observed failure mode that motivates it and prune the hint once that failure stops recurring; keep worked examples when trimming generic instruction — examples are scaffolding the proposer imitates, and the abstract rule alone is weaker. Chain-shape hints: prefer stepped framework upgrades over one-shot leaps; pure JDK upgrades exempt from staging. When the proposer applies the right step shape but consistently picks the wrong target, the gap is information-availability not hint quality — enrich the observation with the specific pom element / classpath dump / source-grep result the proposer needs, rather than enumerating possible targets in the prompt. Pre-recipe pom-XML edit: when `mvn rewrite:run` fails because the unmodified project itself doesn't compile (JDK-removed javax classes still imported, or annotation processors pinned at versions too old for the target JDK), no chain mutation can recover — the fix is a direct pom XML edit ahead of any rewrite step, including elements like `<annotationProcessorPaths>` that OpenRewrite primitives don't reach. Regime change: when the harness, recipe artifact set, or environment is altered in a way that materially changes what counts as a failure, INVALIDATE the trajectories of every stage whose history was conditioned on the old regime; cheapest invalidation is deleting the affected trajectory files so the next pass starts those stages from scratch. When the proposer cannot reach a fix regardless of prompt or observation enrichment (requires a primitive that doesn't exist), ship that primitive deterministically in the harness, not more hints.
   - **Reward:** mean attempts-to-PASS across passing stages; reduction in attempt-waste (dupe-chain submissions, same-failure-signature repeats); fraction of clusters that converge on the first hint-aware attempt vs require iteration.
   - **Repeat:** whenever a new failure cluster emerges, whenever observed proposer waste on a cluster exceeds half of attempts spent, or when a hint stops earning its place (the failure it addressed no longer recurs).
11. **Fitness (proposer investigation depth):** the proposer reaches source-backed conclusions on each stage without paying the context cost of every source-file possibility on every call.
    - **Purpose:** failure observations naming a file and line are useless if the proposer has to guess at the code structure — most failures observed (call-site references to removed superclass methods, annotation duplication across nested classes, mixed-package transitive deps) require knowing what is actually in the cited source, and pre-baked `stage_facts` covers only the patterns triage already knows to look for; without targeted investigation, the proposer either iterates blind to budget or the prompt grows library entries until the entries themselves crowd reasoning.
    - **Contract and constraints:** the proposer's single-call context budget does not scale with repo size — investigation that needs to read source or grep the working tree happens in delegated calls that return digested findings to the proposer, not raw bytes. Investigator capability comes from OpenHands (chosen as the maintained framework that already provides the read/grep/shell tool surface and a sandboxed execution loop) — the harness does not reinvent tool-call iteration, context pruning, or conversation state. Delegated investigators have their own context windows and own tool budgets, are constrained by a system prompt that forbids chain proposal or build mutation, and return only a bounded summary plus a token-bounded raw excerpt around the cited site. Investigators are addressable from the proposer's response schema (the proposer asks for one) and from the harness (when the proposer's chain stops mutating across attempts). Investigator outputs are persisted alongside the trajectory so the operator can audit the chain rationale against the source the investigator surfaced.
    - **Search hints:** patterns the proposer guessed wrong on at iter N are the patterns an investigator should be cheap to reach for at iter N+1; when the proposer keeps repeating the same chain across attempts on the same failure signature, the missing input is structural source detail, not a different recipe; triage already produces a summary at `sha_from` (ff #10's stage_facts feed), so investigators are triggered when the failure observation says something triage didn't. A general-purpose autonomous-agent framework didn't earn its keep as the proposer (proposer's job is bounded chain emission); it earns its keep as the investigator (investigator's job is exactly what such frameworks are designed for: read files, grep, summarize). Don't re-litigate the proposer choice when adopting the framework for the investigator role.
    - **Reward:** fraction of multi-attempt stages where the next chain is materially different from the previous (vs identical-chain bails — ff #1 already counts these); mean number of investigator delegations per failure observation (high mean signals the proposer is over-asking, zero mean signals the proposer is reasoning blind on stages that need source detail); operator-spot-check rate at which the proposer's stated rationale aligns with the investigator's findings.
    - **Repeat:** whenever a stage cluster shows the proposer iterating to budget with identical chains; whenever the operator finds (during spot-check) that the proposer's rationale is contradicted by the source it cites.
