# AGENTS.md

**Per-attempt history:** each `attempt_N/README.md` snapshots the AGENTS.md state that attempt ran under — audit only, not read by the agent.

1. **Problem (writing this file):** keep AGENTS.md compact and outcome-named.
   - **Value:** compact, outcome-named files force the agent to re-derive the *how* each iteration from its tools and the corpus; without this discipline the file rots as the domain shifts and the agent over-fits to one iteration's specifics.
   - **Contract and constraints:** A problem is a self-amplifying attractor — one autonomous concern with a single extremum (its Reward) and a single trigger (its Attention mechanism) that pulls the agent's attention and amplifies until satisfied. Each is a concern the operator has offloaded to the agent (the agent watches, triggers, and drives it), making this file a delegation protocol, not a checklist. Only one problem is in foreground at a time (Ukhtomsky); before every action the agent verifies exactly one is dominant — concurrent problems signal confusion to resolve first. Control passes when a stronger trigger fires or the current extremum is approached; an interrupted problem is resumed, not restarted, so each persists enough state (trajectory files, addenda, in-progress search position) for resumption to be genuine. Problems are written for an intelligent agent searching a fuzzy environment with the available tooling in Ralph loops: supply no implementation detail the agent can fill itself, and — since humans reread this file often — keep each as short as possible. Every problem has five sections: **Value** (why it exists — would the project notice its removal?); **Contract and constraints** (the operator-only boundary the other sections optimize within — the agent does not edit it); **Solution search approach and hints** (what observed failures taught the agent to try first, plus this agent's own loop or trigger for returning to the problem); **Reward** (one extremum the operator reads off without judgement — a count, fraction, or comparable artefact); **Attention mechanism** (how a change this problem makes or notices in the shared environment becomes salient to whichever problem must react — the channel it publishes a signal to or watches, e.g. a log line, a metric, a `feedback.jsonl` entry, or an observability stream — as distinct from the agent's own re-entry loop). The one-liner names what the problem produces, not how. Backtick-pinned concretes (paths, command shapes, file and element names) are recognition scaffolding and survive trims; aging enumerations (versions, models, repo names) get stripped. The file is clustered — meta (P1); recipe and setup (P2–P4); substrate (P5–P10); public distribution (P11); demand-driven application (P12 discovery/routing → P13 single-hop / P14 multi-step) — and within a cluster ordered by what serves what (the production rung-2 is a panel of off-the-shelf agents, so portability and cross-agent robustness are properties of P3, not separate problems). The open attempt is always the directory `current_attempt`; every tool references `current_attempt/...` (never a hardcoded number). An attempt gets its number only at archival: closing it renames `current_attempt` → `attempt_N` and leaves it frozen in place — so opening the next attempt needs no scripts, just that rename (no symlink to move). Attempt-agnostic shared helpers live in a frozen attempt (e.g. `attempt_7/tools`) only while never modified; the moment the open attempt must edit one — e.g. the build/test entry script `run_one_stage_v2.sh` or the `docker_phase`/`shallow_fetch` harness it reaches through — it is copied under `current_attempt/` and referenced sibling-relative (resolve siblings via `__file__`, never a hardcoded `attempt_N` path), so the tuned harness ships and is versioned with the attempt while frozen attempts stay byte-frozen.
   - **Solution search approach and hints:** read → why → intent — for each clause ask "why is this here?". Strip when the answer is mechanism the agent already fills, a bare plea for the rule, or an aged enumeration; keep when it scopes when the rule applies, names a recognition concrete, or signals when to revisit. A topic that always fires inside another problem's loop is mechanism of that loop — it belongs in that loop's Contract, not as its own problem. Re-run every editing pass; additionally, whenever the operator touches a problem's Contract and constraints, the next pass re-audits that problem's agent-mutable sections.
   - **Reward:** cuts that lose words without losing the rule or its scope; after the pass, every problem's Reward names something the operator can read off without judgement.
   - **Attention mechanism:** the file is the channel — an operator edit to a problem's Contract, or a problem's agent-mutable sections visibly bloating, is the signal that pulls this meta-problem to re-audit that problem.

## Recipe cluster

2. **Problem (Java LTS bump artifact for any coding agent):** ship one `bump-java-version` skill — a portable skill package consumable by any tool-using coding agent (no assumption of our harness, scratch layout, or runtime) — that on each next-LTS bump (8→11, 11→17, 17→21) drives the highest corpus PASS rate.
   - **Value:** an empirical migration skill that beats the one-shot OpenRewrite baseline is the primary research goal; portability across agents is the audience constraint, so the deliverable is self-contained and harness-agnostic.
   - **Contract and constraints:** the deliverable is the `bump-java-version` skill — a single self-contained `SKILL.md`: a standard-tools-only hand manual (JDKs, Maven, and OpenRewrite recipes pulled from Maven Central; no project-specific scripts) that migrates a Maven project one Java LTS step so it compiles under the new JDK and every test that passed before still passes; loadable by any tool-using coding agent that supports skills (no assumption of our harness, scratch layout, or runtime), all stage-agnostic (stage facts arrive via the consumer's harness as a prepended Stage header). Reads P4 for the stage list. Per-stage validation is delegated to P3, which returns `{terminal_rung, terminal_outcome, artifact_updates}`; fold `artifact_updates` into the corpus `bump-java-version` skill iff that does not regress corpus reward, else leave it unchanged and record the stage unintegrated. Concurrency across stages from P9.
   - **Solution search approach and hints:** when a failure family's stages only pass at the strongest rung (Opus) and won't harden down to the agent panel, that family lacks a corpus-level remedy — commission a recipe via P3's search or add a failure-output row. An `artifact_updates` that passes its own stage but regresses others on rollout is over-fit — generalise its trigger or scope the fix narrower. Loop continuously round-robin until reward stops improving, restarting from cached trajectories whenever the corpus artifact's sha materially changes.
   - **Reward:** corpus PASS rate (fraction of stages P3 terminates at the agent panel with `terminal_outcome=PASS`) vs the one-shot `UpgradeToJava<jv_to>` baseline.
   - **Attention mechanism:** `feedback.jsonl` is the channel — a failure family clustering there, or its stages only passing at the strong end of the rung ladder (stuck at Opus), signals that the family needs a corpus-level remedy.
3. **Problem (per-repo skill + recipe escalation):** for each (repo, commit) stage P2 hands down, harden the current best (skill, recipes) artifact until it drives the bump to PASS or labelled BAIL across a Qwen-driven panel of off-the-shelf agents — by making the stage pass on a ladder of executors, all driven by Claude — routinely two rungs: first Claude+Opus 4.8, then Claude managing the production panel — the same Qwen 27B run headless through three unrelated third-party agents (`opencode`, `kilocode`, `openhands`) in one unified harness (`agent_drive_one.sh`, one image) where the agent is the only variable — with Claude patching the artifact at the stronger rung so it survives the weaker one. The panel replaces both the old single-agent (OpenHands) target and the separate portability + cross-agent witnesses: three stranger agents on the identical skill make portability inherent, and agent disagreement on a stage is the sharpest skill-clarity signal short of Opus.
   - **Value:** P2 sees corpus PASS/FAIL rates but cannot distinguish "stage genuinely unfixable", "production LLM too weak", "production harness wrong", and "artifact has a gap". This problem separates those for one stage: the routine rungs name most of the failure mode — fails even at Opus → genuinely unfixable or needs a new recipe; reaches Opus but fails the panel → the skill is unclear to a weak LLM or leans on one agent's tool-use conventions. The panel itself resolves the last ambiguity that used to need a separate bare-Qwen probe: fails under all three agents → the skill needs hardening; fails under one but passes the others → that agent's harness limit (surface to P8), and the cross-agent disagreement names the exact instruction to tighten. Because the three agents are off-the-shelf and run the skill with no assumption of our harness, the panel also witnesses portability (P2's no-assumption rule: it must pass under strangers' agents, not only inside one) — so a separate portability/cross-agent witness problem is no longer needed. It is also the path by which a single stage's skill/recipe insight propagates into the corpus artifact.
   - **Contract and constraints:** The (skill, recipes) artifact has a pinned shape so every rung reads the same structure and a failure maps to a known section. The skill is a single self-contained `SKILL.md` — a standard-tools-only hand manual (JDKs, Maven, and OpenRewrite recipes from Maven Central; no project-specific scripts) — whose numbered sections are pinned in order: Tools; record the baseline under `jv_from`; make Lombok safe; run the OpenRewrite migration (recipes by FQN); apply the deterministic JDK-removal fixes as pom edits; compile + test + conserve under `jv_to`; the Spring Boot upgrades a Java bump forces; a Troubleshooting table (failure → remedy); and When to bail (the live instance is at `.agents/skills/bump-java-version/SKILL.md`). Two pins a reader needs here: the Troubleshooting table's fix vocabulary is {pom edit, `mvn rewrite:run` by FQN, docker cleanup, labelled bail, ignore-with-justification}, and the Reward (PASS) is `mvn compile` succeeds under jv_to AND pre-pass test set ⊆ post-pass test set. Recipes are whatever the manual names by FQN, resolved from Maven Central (public OpenRewrite recipes); there is no bundled recipe catalog. Stage facts (repo URL, sha, jv_from, jv_to, workdir) are injected by each rung's harness as a prepended Stage header, never in the `SKILL.md`, so the skill stays portable. The search is one schema: per stage, the **search results** — the artifact version run at each rung (`opus`, then the agent panel `opencode`/`kilocode`/`openhands`) with its outcome — and the one **best search result**, the hardened artifact that passes the panel. Claude is the sole searcher at every rung: it runs the current artifact under that rung's executor and, on failure, patches the artifact (rewrite any `SKILL.md` section, add or reorder Troubleshooting rows, add a recipe FQN, tighten or reorder a step); the executor (Opus 4.8, or the Qwen 3.6 27B FP8 proposer — served from P7's `/qwen-3.6-27b-fp8/v1` route on GPU0, `max-model-len` 262144 (256K) — run through any panel agent) only runs what Claude ships; the panel runs all three agents on the IDENTICAL skill+env+scoring (the unified driver, one image), the agent being the only variable. The routine ladder goes Opus → panel (strong → production): Opus surfaces "genuinely unfixable / needs a new recipe"; the panel is the production target the artifact must reach under every agent; a fix made at Opus is re-validated on the panel, since it can regress. A rung's FAIL is re-run-confirmed (flakes re-run at the same rung, not escalated); the stage terminates when the panel reaches PASS under all three agents or a labelled BAIL. Per (stage × rung × validation-run) the harness appends one line to `current_attempt/feedback.jsonl` (`rung ∈ {opus, opencode, kilocode, openhands}`, `outcome ∈ {PASS, BAIL:<label>, FAIL}`) and, before invoking a rung, snapshots the exact `SKILL.md` to `skill_snapshots/<sha>.md` (content-addressed, idempotent); the full per-stage record is `per_repo_iter/<slug>/trajectory.json` alongside `oh_dialogue.log` and `bump.diff`. Each rung's session is isolated (own workdir, conversation id, surefire path); concurrency bounded by P9. Output to P2: `{stage, terminal_rung, terminal_outcome, artifact_returned}`, where `terminal_rung` is the weakest rung reached at PASS or labelled BAIL (ideally the full panel) and `artifact_returned` is the hardened artifact; P2 decides folding.
   - **Solution search approach and hints:** fast-fail at a rung — if Claude+executor stalls (no progress for >N actions on the same step) treat it as FAIL at that rung, don't spin; never recurse on flakes (one-off network error, Docker glitch) — re-run the same rung. On a FAIL, Claude's first probe is to read the executor's actual trace (the agent transcript) to find where its reasoning diverged from the skill — typically a missed trigger paraphrase, a skipped step, or a tool-use convention the `SKILL.md` didn't anticipate. Claude's patch is the smallest possible (a one-sentence tightening, a row reword, a step reorder), not a wholesale rewrite; add a new recipe only when no prose change could fix the executor's behaviour. After patching at Opus, re-validate on the panel, since a fix can regress. A stage that keeps failing the panel after patching: if it fails under all three agents it's a genuinely unclear skill — harden it; if it fails under one agent but passes the others, that's that agent's harness limit (surface to P8, not a skill rewrite), and the cross-agent disagreement names the diverging instruction to tighten. Portability drift (the skill coming to depend on one agent's scaffolding) shows up automatically as a widening cross-agent gap, so no separate portability probe is needed. Re-invoke per stage each pass from P2, and again when the corpus artifact's sha changes and the stage previously did not pass the panel.
   - **Reward:** fraction of input stages where the agent panel reaches PASS or labelled BAIL — both initially and after each patch; the metric is "panel success rate", not "any-rung success rate". Secondary: average patches per stage to reach panel success (lower is better).
   - **Attention mechanism:** `feedback.jsonl` is the channel — a re-run-confirmed FAIL trajectory written there is the signal that pulls Claude to patch the artifact at that rung; the panel still failing after patches (under all agents) signals P8 (a structural harness limit), not another skill rewrite.
4. **Problem (dataset = repo list + per-run randomized-baseline sampler):** maintain `dataset-repos.json` (repo names only); draw the eval set fresh each round by sampling, per repo, a random commit that declares Java 8/11/17 AND compiles under that JDK.
   - **Value:** a fixed-sha corpus lets P2 (and its GEPA/EvoSkills optimizers) overfit to specific commits, and biases toward already-prep'd states that score high without real migration work. Sampling a *different* valid baseline per run makes the eval a moving target — only improvements that generalize across each repo's history survive. It also retires baseline curation: validity is checked when a baseline is drawn, not maintained as a clean-list.
   - **Contract and constraints:** the persistent dataset is `dataset-repos.json` (a list of `owner/repo`). Each round, `tools/sample_shas.py --seed <round>` emits `dataset-shas.json` (committed, lives in the attempt folder) — **N = 100 datapoints/attempt** for now, a seeded random subset of `dataset-repos.json` (the count rises as we move toward GEPA) — of entries `{repo, sha, jv_from, stars}` where `jv_from ∈ {8,11,17,21}` and the bump target is always the next LTS (8→11, 11→17, 17→21, 21→25) — **derived from `jv_from`, not stored**; the `sha` is **verified to test-compile under `jv_from` at sample time**, and `stars` is the repo's GitHub stargazer count (annotated via the GraphQL API after sampling; `-1` if the repo is unresolvable). One repo can yield several baselines — one per Java version in its history (`--multi`). Repos are sampled **most-stars-first** (the sampler's repo list is star-sorted before sharding), so a bounded run keeps the highest-signal baselines. P2 reads the per-run file. A different `--seed` ⇒ different shas; the same seed is reproducible. A repo that yields no compiling 8/11/17/21 baseline within the attempt bound contributes nothing that round — never a false `baseline-broken` into P2. Junk (already ≥ `jv_to`, no pom, non-compiling) cannot enter the dataset because the sampler rejects it.
   - **Solution search approach and hints:** bounded rejection search per repo — shuffle commits with the run seed, walk them, cheap-reject (no pom / jv ∉ {8,11,17}) for free, and `test-compile` each eligible candidate under its `jv_from`; accept the first that compiles, capped at ~10 compile attempts (plus a scan cap on commits inspected). `test-compile` (main+test) is the validity bar so P2 never inherits a baseline whose tests don't even compile. Sampling across **full** history organically surfaces 8→11 and 11→17 hops, not just 17→21 — fetch unshallowed so the walk sees real ancestry. Keep `dataset-repos.json` broad (distinct owners, dependency-family spread); for a repo that rarely yields a baseline, raise `--max-attempts` rather than hand-curating. Use a fresh seed per optimization round and score edits on the **mean across seeds**, discarding seed-specific gains.
   - **Reward:** sampler hit-rate (fraction of `dataset-repos.json` that yields a valid compiling 8/11/17 baseline per run) and the LTS-hop diversity of the sampled set; and, for the optimizer, **low variance of P2's score across seeds** — a genuinely general artifact scores about the same on any sampled baseline set, an overfit one does not.
   - **Attention mechanism:** the per-run `dataset-shas.json` and the spread of P2's score across seeds are the channel — a repo that never yields a baseline (raise its attempt bound or drop it from `dataset-repos.json`), or a P2 gain that evaporates under a new seed (an overfit edit — reject it), is the signal that pulls P4 to adjust sampling.

## Substrate cluster

5. **Problem (proxy dependency resolution):** every build's external-artifact resolution goes through a local caching proxy with plural upstream mirrors.
   - **Value:** without a caching proxy, build failures get attributed to upstream availability blips rather than code state, polluting the parent loop's reward signal.
   - **Contract and constraints:** the proxy caches every artifact it serves and survives across iterations. Consumers (P2, P4) attribute build failures to code state, so an unresolved artifact triggers a proxy widening before that build counts toward reward.
   - **Solution search approach and hints:** include both live and archival mirrors; container builds reach the proxy by container-network DNS, not host IPs; when a build fails on "cannot resolve X", widen the upstream set before treating the failure as code-attributable. Re-enter whenever a parent loop's failures cluster on artifact resolution.
   - **Reward:** per-artifact cache-hit ratio (higher is better; resolution failures become distinguishable from compile failures in the parent loop's classification).
   - **Attention mechanism:** build logs are the channel — a `cannot resolve X` line is the signal that pulls P5 to widen the upstream set before that failure is counted as code-attributable.
6. **Problem (maintain local environment):** known stable paths, scratch-space discipline, and disk/cache stewardship for every other problem.
   - **Value:** iteration outcomes must vary only with what's being measured; unstable paths, shared scratch, and contention drift the noise floor up, and one problem exhausting a shared resource crashes another and corrupts the parent loop's results.
   - **Contract and constraints:** project workdir at `$HOME/java_8_11_17_to_java_21` (the one path pinned here — every other is an env var resolved from `.env` so the value lives next to credentials). Env vars consumers read: `MIRROR_CACHE_DIR` (per-repo git mirrors as `<owner>/<repo>.git`); `SCRATCH_ROOTS` (colon-separated writable mounts by preference; agent picks the first with headroom); `SCRATCH_SUBDIR_TEMPLATE` (relative template per scratch dir, e.g. `tmp/<repo_name>/<ff_specific_subdir>/`). All build toolchains and recipe execution run inside Docker; SSH calls to the work host share one session, not one per command; free host resources held by unrelated work when this project needs them. **Docker container logs are rotated** — the daemon's default `json-file` driver has no rotation, so an unbounded container log (one reached 55 GB here) silently fills the disk and crashes dependent services (e.g. P10's compactor on its next write); `observe_rotate.py` truncates over-cap container json-logs via a root container, and the permanent fix is docker daemon `log-opts` (`max-size`/`max-file`) applied on a maintenance restart. A problem declares a scratch-dir name; this problem resolves it against `SCRATCH_ROOTS` + `SCRATCH_SUBDIR_TEMPLATE`. Each run reaps its own scratch workdir on completion (in a `finally`); because the build container writes root-owned files, the reap runs through a root container, not host `rm`. A scratch dir that outlives its run is a reap failure to investigate.
   - **Solution search approach and hints:** when a pattern repeats and is slow, pin a known location or cache; when a pattern risks an external rate-limit, cache the upstream once; never let docker-owned root-rw files leak — each run cleans its own scratch through a root container (not host `rm`) in a `finally`, so cleanup is never skipped or deferred. Re-enter whenever a new noisy or slow pattern emerges, or when a run's scratch survives the run — that means the per-run reap failed, so fix the reap.
   - **Reward:** iteration variance from non-measured factors approaches zero — zero ban incidents, predictable per-iteration wall-clock, locations re-used across iterations, zero disk-full crashes, bounded scratch-dir count per repo (a leak triggers investigation, not just cleanup).
   - **Attention mechanism:** host metrics and scratch-dir counts are the channel — a scratch dir that outlives its run, or a per-repo scratch-dir count creeping up, signals that the per-run reap failed and P6 must fix it.
7. **Problem (vLLM spin-up):** stand up tool-capable OpenAI-compatible chat-completion endpoints callable from inside Docker, with authentication enforced.
   - **Value:** P2's agent backend and P10's compactor cannot run without a tool-capable model endpoint reachable from inside docker containers; this problem owns that contract surface.
   - **Contract and constraints:** all endpoints in active use serve models from the same generation family — never mix generations, since prompts calibrated against one drift on another. When two consumers have meaningfully different throughput/depth tradeoffs they get separate endpoints on separate accelerators so they don't contend for slots. A single public gateway (`inference.mikhailov.tech`, a Caddy reverse proxy) fronts both endpoints — `/qwen-3.6-27b-fp8/v1` reaches the FP8 proposer (GPU0) and `/qwen-3.6-35b-a3b-awq/v1` the AWQ observability compactor (GPU1, a Qwen 3.6 35B-A3B MoE) — the path names the served model; one hostname and credential, so consumers never hardcode internal container IPs (which churn on daemon restart), and unmatched paths return a 404. Per-consumer model identity and call-time options (thinking mode, etc.) live in the consumer's own Contract.
   - **Solution search approach and hints:** arrive at a working endpoint set through a ralph loop over container, model, and reverse-proxy config; converge by what consumers can actually call, not what the endpoint exposes in isolation. Re-enter on any consumer reporting degraded service.
   - **Reward:** consumers report uninterrupted service.
   - **Attention mechanism:** consumer call results are the channel — a consumer logging a degraded or failed completion is the signal that pulls P7 to re-check that consumer's endpoint.
8. **Problem (agent runtime spin-up):** stand up the configured agent runtime as a credentialed service reachable from the work host, with conversation lifecycle, event-stream emission, and a context-management primitive configured.
   - **Value:** P2's recipe search needs an agent runtime that accepts a prompt, executes against a working tree, emits structured events the harness consumes, and survives the deepest stages without losing trajectories to context overflow; this problem owns that surface.
   - **Contract and constraints:** the runtime exposes a callable interface (in-process SDK, HTTP API, or MCP socket) reachable from the harness, with tool invocation, conversation state across iterations, and event streaming. It is configured with at least one context-management primitive — an automatic condenser that folds older events into a summary at a threshold, or delegated subagents with their own context windows — so single-call context budget does not scale with repo depth and overflow is structural, not catastrophic. Every exposed surface shares P7's credential discipline: no unauthenticated bind, no credentials in trajectory state or in scratch dirs reaped by P6. Sub-context activity is captured in the event stream (P10) so the operator can audit when context was folded. The runtime has network reach to P5's caching proxy (same docker network or equivalent route) so every Maven invocation resolves through the cache, not direct upstream — without this, wall-clock balloons and upstream hosts ban us. Runtime identity is pinned per attempt so P2's PASS rates are comparable across passes. The runtime accepts N concurrent isolated sessions (own conversation id, workspace, event-stream slice), N dictated by P9; sessions share the LLM endpoint and proxy but never a workspace.
   - **Solution search approach and hints:** prefer maintained frameworks over reinventing tool-call iteration, context pruning, or conversation state; choose a runtime by what P2 can actually drive, not by what it advertises. Network reach goes through the same reverse-proxy + auth surface as the vLLM endpoints; in-process SDK use is permitted but not exempt from that credential discipline. If the agent stalls or repeats actions on the same failure across iterations, the missing input is usually structural source detail that needs sub-context — not a different prompt. Re-enter on any P2 stage reporting an agent-runtime failure unrelated to prompt or repo state, whenever context overflow drops a trajectory, or whenever a stage cluster shows the agent stalling without obvious cause.
   - **Reward:** P2 reports uninterrupted agent execution; fraction of stages where context overflow caused trajectory loss approaches zero; zero credentialless surfaces.
   - **Attention mechanism:** the agent event stream is the channel — a trajectory dropped to context overflow, a stall repeating actions on one failure, or an unauthenticated surface, signals P8 to re-examine the runtime.
9. **Problem (runner saturation):** keep every host compute resource (CPU, memory, GPU compute, GPU memory, disk I/O, network) in a healthy utilisation band while any parent loop is making progress; the binding constraint shifts across the loop's phases and is re-detected, not assumed.
   - **Value:** an idle host wastes budget; any over-saturated resource drops the loop's wall-clock-per-PASS reward to noise. The binding constraint is rarely CPU alone — under N-way agent parallelism it shifts from CPU at startup to GPU compute during agent rounds to disk/network during mvn cache warming to GPU KV-cache memory under sustained batching.
   - **Contract and constraints:** composes with the parent loop rather than standing alone — saturation decisions are conditional on which loop is active, its recent action history, and which resource is nearest its band edge; any concurrency dial the agent can reach is in scope. The primary output is a per-loop concurrency recommendation N (read by P2 to size its per-pass session fan-out into P8's runtime); N may shift mid-pass as the binding constraint shifts.
   - **Solution search approach and hints:** sample every relevant resource periodically (host CPU + load avg, host memory, per-GPU utilisation + memory + KV-cache from the vLLM logs, disk I/O on the work mount, network to P5's proxy and P7's endpoint, docker daemon health); learn the band per resource; re-detect the binding constraint when the loop changes phase. GPU compute-percent snapshots are bursty under batching — read the average over a window, not the instant. Run continuously, dampened against oscillation; pause when no parent loop is active.
   - **Reward:** sustained healthy band across every relevant resource without thrashing or stalling the parent loop; the binding constraint correctly identified at any moment.
   - **Attention mechanism:** the resource samples are the channel — any resource (CPU, GPU compute/KV-cache, disk I/O, network, from `host_metrics` + the vLLM logs) nearing its band edge signals P9 to re-detect the binding constraint and re-issue N.
10. **Problem (frog's eye — compact observations):** everything on the host writes to `/var/log`; Vector ships the tree to a compacting model that condenses it into the orchestrator-visible digest.
   - **Value:** raw observation streams exceed the orchestrator's context budget; without a faithful digest, decisions are made from the wrong fraction of the data and trust in the digest erodes silently.
   - **Contract and constraints:** capture is file-glob based, auto-discovering — no per-source or per-container wiring. A single Vector instance (`obs-vector`) globs two trees: **`/var/log/**/*`** (host system logs + project process logs, written under `/var/log`, e.g. `/var/log/observe/app/`, never `/tmp` scratch) and **`/var/lib/docker/containers/**/*-json.log`** (every container's stdout via the json-file driver — new containers appear as new files automatically, no daemon change and no Docker API dependency; the json envelope is unwrapped to the bare message). Plus two host-resource streams: Vector's `host_metrics` (CPU/mem/disk/load) and a GPU sampler (`nvidia-smi` → `/var/log/observe/app/gpu.log`, picked up by the same glob). Vector ships all of it to the compacting model — Qwen 3.6 **35B-A3B AWQ** (a 4-bit MoE, ~3B active/35B total — chosen over the dense 27B for far higher condense throughput) via P7's `/qwen-3.6-35b-a3b-awq/v1` (`chat_template_kwargs.enable_thinking: false`; context budget = that endpoint's `max-model-len`, currently 64K; input over budget is chunked and merged, never truncated). **Repetitions are collapsed before the model** — events are grouped by a signature (stream + source + message with numbers/ids/timestamps masked) and each group is sent once as an exemplar + a count, so a high-frequency repetition (container/veth churn, recurring warnings) condenses ~100× instead of being re-sent line-by-line. This collapse is what keeps the condense rate ahead of a high-volume firehose; without it the model drowns re-reading identical lines. The compactor writes the orchestrator-visible digest. The digest is **multi-resolution** — the compactor presents several time windows (short to long, e.g. seconds to tens of minutes) side-by-side so an anomaly shows by window-to-window contrast, not by absolute level. Capture is continuous (runs whether or not anyone is reading it) and **never content-filtered or throttled**: noise is fixed at its source (a crash-looping unit is stopped, not excluded from the stream) so the digest reflects ground truth; only unreadable files are skipped (binaries, compressed/rotated logs, the journal binary dir). Nothing Vector or the compactor writes may land inside the watched tree (no self-ingest). Every log and spool is size-bounded so a full disk cannot silently stall the digest; both the capture layer and the compactor self-heal (auto-restart). The compactor is never load-bearing for an irreversible decision without a spot-check against the raw `/var/log`. Any problem execution whose run window overlaps a gap in capture is invalid.
   - **Solution search approach and hints:** Vector is the canonical capture layer (`obs-vector` container; config versioned at `current_attempt/tools/vector.toml`, never `/tmp`). Container logs are captured by globbing the json-file driver's files (`/var/lib/docker/containers/**/*-json.log`, mounted read-only into `obs-vector`) — file-based, so every container is auto-discovered with no API dependency and no daemon log-driver change (the `syslog` driver was rejected: it would break the `docker logs` API and rate-limit under load; keep `json-file` + its `log-opts` rotation). Add a new stream by writing it to a file under one of the globbed trees, not by extending the compactor. Emit only when state changes materially since the last digest (a quiet stream stays quiet); surface failure signals, not just happy-path; on input over budget chunk-and-merge rather than drop; periodically spot-sample raw vs digest to recalibrate trust. Re-enter whenever a new noisy source appears or the digest goes stale.
   - **Reward:** the **condensation ratio** — raw events in : distinct/anomalous entries out — at a near-zero miss rate. The binding metric is the *condense rate* (distinct signal the model must absorb per second), **not** raw inflow: because repetition collapses to an exemplar + count, a high-volume but repetitive firehose (hundreds× ratio) is cheap, while novel/anomalous signal is always carried. The eye falls behind only when distinct signal — not volume — outruns the model; if it lags, raise the ratio (collapse harder / fix the noise source), don't just demand a faster model. Quiet and repetitive streams stay quiet so emitted entries carry signal.
   - **Attention mechanism:** the digest is the channel — a materially new pattern in the `/var/log` stream (a new error signature, a build or inference failure, a throughput stall, a hardware/OOM/Xid event, a bail label) becomes salient by contrast across the digest's time windows (short to long) and against the prior digest, and is emitted; steady state stays silent. If a source that matters is not writing into `/var/log`, the frog's eye is blind to it — the fix is always to point that source at `/var/log`, never to special-case it in the compactor.
